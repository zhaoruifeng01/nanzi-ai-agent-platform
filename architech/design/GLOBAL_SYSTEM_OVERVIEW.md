## 3. 整体系统视图

从公司整体视角来看，当前围绕数据中心智能运营大致包含 **四大核心系统**，本仓库主要覆盖的是第 2、3、4 点中的一部分：

1. **南孜・数据底座（NanZi Data Foundation，数据采集与大数据加工平台）**
   - 负责采集 YES、NC、动环等多源数据：设备、告警、能耗、环境等。
   - 在大数据平台内部完成：清洗、聚合、建模特征等加工，形成统一数据资产。
   - 对外通过 API / 表视图 / 实时流等接口，输出加工后的指标、明细、聚合结果。
   - 本项目 **不负责** 该平台的详细设计与开发，只作为其 **数据消费者** 使用其输出。

2. **南孜・数据服务平台（NanZi Data API Platform，技术名 `api_service`，独立通用数据服务层）**
   - 定位为 **通用 Data API 平台**，对上屏蔽底层存储的差异，对下统一封装多种数据源：
     - ClickHouse（OLAP 分析）
     - HBase（实时/宽表）
     - MySQL（业务维度、元数据等）
   - 主要职责：
     - 将大数据平台加工后的结果 + 各类在线库，统一服务化为 REST/JSON API。
     - 实现统一的认证鉴权、角色层 + 数据资源层（region/room 等）权限控制。
     - 提供统一的响应 Envelope：`{ code, message, data, timestamp, trace_id }`，对齐 OpenSpec 规范。
     - 提供查询编排与多数据源路由能力，作为后续所有应用和 AI 的“取数中枢”。

3. **南孜・智能体平台（NanZi AI Agent Platform，独立的 AI 服务系统）**
   - 定位为 **通用 AI Agent 平台**，不是南孜 Web 的内嵌功能，而是独立服务：
     - 提供 ChatBI（自然语言取数 + 图表）能力；
     - 后续扩展 RAG 知识库、多 Agent 意图识别、自动化工作流等。
   - 主要职责：
     - 接收用户自然语言请求，进行意图识别、NL2SQL 转换、查询编排；
     - 优先通过 `api_service` 获取数据（不直接贴 ClickHouse/HBase/MySQL），复用统一权限与审计；
     - 按“**主图表 + Markdown 摘要 + 可选 Widgets（如 SOP 卡片）**”的格式返回结果；
     - 通过公司内部私有化部署的、兼容 OpenAI 协议的模型网关调用大模型。

4. **南孜应用端（南孜・运营控制台 + 南孜・业务服务，Web / 移动 + 自有后端）**
   - 包括：
     - 南孜・运营控制台（NanZi Ops Console，本仓库 `frontend-app`）；
     - 南孜・业务服务（NanZi App Service，待开发，将独立成后端工程）；
     - 未来的移动端 App / H5 等。
   - 南孜・业务服务主要职责：
     - 面向具体业务场景（大屏、资源管理、告警管理等）聚合 API，做领域逻辑；
     - 管理自身业务数据库（如用户、权限、配置、看板布局等）；
     - 调用 南孜・数据服务平台（api_service） 获取各类监控 / 统计数据；
     - 调用 南孜・智能体平台（AI Agent Platform，ChatBI / 助手）提供智能取数与问答能力。

## 4. 系统间关系与调用拓扑

### 3.1 高层关系图

用一个简化的“自上而下”视图描述整体关系：

```text
          ┌──────────────────────────┐
          │        最终用户          │
          │   Web / 移动端操作人    │
          └─────────▲────────────┘
                    │ 交互 (浏览器 / App)
                    │
          ┌─────────┴────────────┐
          │   南孜应用前端 (Vue)   │  ← 本仓库 frontend-app
          └─────────▲────────────┘
                    │ HTTP / WebSocket
                    │
          ┌─────────┴────────────┐
          │   南孜应用后端 (待开发)│  ← 业务聚合 / 会话 / 权限
          └──────┬────────┬───────┘
                 │        │
     调用数据 API│        │调用 AI 能力
                 │        │
     ┌───────────▼───┐   ┌───────────────▼────────────┐
     │  api_service   │   │   AI 智能体平台 (AI Agent) │
     │ 通用数据服务层 │   │ ChatBI / RAG / 多 Agent 等 │
     └───────▲───┬───┘   └───────────────▲────────────┘
             │   │                       │
      读取加工后│                       │ 查询数据 / 追加分析
        数据等  │                       │
             │   │                       │
   ┌─────────▼───┴─────────┐
   │ 数据采集与大数据加工平台 │  ← YES / NC / 动环等
   │  (现有大数据系统)       │
   └────────────────────────┘
```

- **南孜・数据底座** 负责“采 + 算”；
- **南孜・数据服务平台（api_service）** 负责“统一对外提供可消费的数据 API”；
- **南孜・智能体平台（AI Agent Platform）** 在此基础上增加“自然语言入口 + 智能分析”；
- **南孜・运营控制台 / 南孜・业务服务** 负责具体业务场景和用户体验，是“消费方”。

### 3.2 典型调用链路示例

#### 场景 A：仪表盘页面直接取数

1. 用户打开南孜 Web 的某个监控/报表页面；
2. 前端发起请求到南孜后端（如 `/api/dashboard/overview`）；
3. 南孜后端根据业务逻辑：
   - 从自身业务库读配置（当前用户可见的机房、视图布局等）；
   - 拼装参数调用 `api_service`（如 `/api/v1/data/temperature/summary`）；
4. `api_service` 根据用户身份与资源范围（角色 + region/room）构造查询，访问底层数据源；
5. 结果返回给南孜后端 → 经过适配/聚合后返回给前端 → 前端渲染图表和指标。

#### 场景 B：ChatBI 自然语言取数

1. 用户在南孜 Web 的 AI 助手输入问题，例如：“帮我看一下华东最近一周的温度趋势”；
2. 前端将问题发送给南孜后端（带上用户会话与身份信息）；
3. 南孜后端调用 **AI 智能体平台** 的 ChatBI 接口（如 `/api/v1/chatbi/query`）；
4. AI 平台内部流程：
   - 使用 NLU / LLM 对自然语言做意图识别与结构化解析；
   - 调用 NL2SQL/查询编排模块，优先生成对 `api_service` 的调用方案；
   - 通过 `api_service` 执行实际查询（自动带上用户角色与资源范围限制）；
   - 基于查询结果生成 ECharts 配置 + 摘要 Markdown + 可选 widgets；
5. AI 平台将结果以统一结构返回给南孜后端 → 前端根据结构渲染：
   - 主图表（趋势曲线等）；
   - 上方/下方的文字结论；
   - 可能的 SOP 推荐卡片等。

# 5. 工程视角与后续扩展

- 当前仓库的 `frontend-app/` 为 **南孜 Web 前端工程**，主要消费：
  - 未来南孜后端对外提供的业务 API；
  - AI 智能体平台返回的 ChatBI 结果结构（经后端适配）。
- `doc/design/` 下的 `api_service_*` 与 `AI_AGENT_*` 设计文档，分别指导：
  - 通用数据服务层 `api_service` 的实现；
  - 通用 AI 智能体平台（ChatBI 为首期功能）的实现。
- 整体架构已经预留：
  - `api_service` 支持 ClickHouse / HBase / MySQL 等多数据源；
  - AI 平台支持未来的 RAG 知识库、多 Agent、自动化工作流；
  - 南孜 Web / 移动端作为 **前端展示层**，可以在不影响后端架构的前提下演进 UI 和交互。

## 6. Git 仓库拆分与命名规范

为支持多团队并行开发和清晰的权限管理，本项目采用 **Multi-repo（多仓库）** 架构，各系统独立维护代码仓库。

### 6.1 仓库列表

#### 服务端仓库（3 个）

1. **yovole-yunshu-api-data-platform**
   - 系统：南孜・数据服务平台（NanZi Data API Platform）
   - 技术栈：Python + FastAPI
   - 职责：多数据源统一 API 封装、权限控制、查询编排
   - 负责团队：API 服务团队

2. **yovole-nanzi-ai-agent-platform**
   - 系统：南孜・智能体平台（AI Agent Platform）
   - 技术栈：Python 3.11 + FastAPI + AgentScope
   - 职责：ChatBI、RAG 知识库、多 Agent、工作流
   - 负责团队：AI 智能体团队

3. **yovole-yunshu-dataweb-backend**
   - 系统：南孜・业务服务（后端）
   - 技术栈：Python + FastAPI
   - 职责：业务场景聚合、会话管理、用户权限、配置管理
   - 负责团队：后端开发团队

#### 前端仓库（1 个）

4. **yovole-yunshu-dataweb**
   - 系统：南孜・运营控制台（前端）
   - 技术栈：Vue 3 + Vite + ECharts
   - 职责：用户界面、数据可视化、AI 助手交互
   - 负责团队：前端开发团队
   - 备注：即当前仓库，后续将 `doc/` 迁移到共享文档仓库后，专注于前端代码

#### 共享文档仓库（1 个）

5. **yovole-yunshu-docs**
   - 内容：跨系统设计文档、接口契约规范、架构图、API Schema
   - 结构：
     - `design/` - 系统设计文档（现 `doc/design/` 的内容）
     - `plan/` - MVP 计划与资源规划
     - `specs/` - 接口契约（OpenAPI、JSON Schema 等）
     - `docker-compose/` - 本地联调环境配置
   - 权限：所有团队 Developer 权限，项目经理 Maintainer

### 6.2 命名规范说明

- **平台类系统**：使用 `-platform` 后缀，体现通用平台属性（`api-data-platform`、`ai-agent-platform`）
- **应用类系统**：使用 `dataweb` 前缀，表示属于"南孜数据运营"产品家族（`dataweb`、`dataweb-backend`）
- **连字符使用**：多个单词之间用 `-` 连接（如 `api-data-platform`），保持一致性

### 6.3 协作机制

- **接口契约管理**：在 `yovole-yunshu-docs` 仓库维护 OpenAPI/JSON Schema，作为各系统的接口规范
- **本地联调**：通过 `yovole-yunshu-docs/docker-compose/` 提供一键启动开发环境
- **文档同步**：各仓库通过 Git Submodule 或 CI 自动同步引入 `yovole-yunshu-docs`
- **权限隔离**：各团队对自己负责的仓库拥有 Maintainer 权限，对其他仓库仅 Reporter（只读）
