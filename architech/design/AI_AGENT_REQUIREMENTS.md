## 2. 项目概述

### 2.1 项目背景
当前平台的数据查询主要通过技术手段实现，业务人员需要通过API或特定工具才能获取所需数据。为了提升业务人员的数据获取效率，需要构建AI智能体，支持自然语言查询数据并以可视化方式展示。

### 2.2 项目目标
- 构建自然语言到 SQL 的转换能力，为业务和运维人员提供“会说话的取数接口”
- 实现 ChatBI 功能，支持在多数据源（ClickHouse、HBase、MySQL 等）之上的自然语言数据查询
- 提供“图表优先 + 短文本总结”的可视化展示能力
- 以通用 AI 服务形态对外提供接口，云枢 Web AI 助手只是其中一个前端客户端
- 支持移动端访问和后续多产品线集成

### 2.3 项目范围
- 自然语言处理引擎
- NL2SQL转换模块
- 数据可视化模块
- AI助手集成
- 移动端适配

## 3. 需求分析

### 3.1 功能性需求

#### 3.1.1 自然语言理解
- **REQ-NLU-001**：支持中文自然语言输入
- **REQ-NLU-002**：能够理解业务术语（如温度、湿度、PUE等）
- **REQ-NLU-003**：支持时间维度理解（如"上周"、"最近一个月"）
- **REQ-NLU-004**：支持空间维度理解（如"华东区域"、"A机房"）
- **REQ-NLU-005**：支持聚合操作理解（如"平均值"、"最大值"、"求和"）

#### 3.1.2 NL2SQL转换
- **REQ-NL2SQL-001**：将自然语言转换为标准SQL查询
- **REQ-NL2SQL-002**：支持多表关联查询
- **REQ-NL2SQL-003**：支持复杂条件筛选
- **REQ-NL2SQL-004**：支持时间序列查询
- **REQ-NL2SQL-005**：支持聚合统计查询

#### 3.1.3 数据查询与处理
- **REQ-DATA-001**：通过 `api_service` 访问 ClickHouse 数据源，不直接贴库
- **REQ-DATA-002**：支持后续通过 `api_service` 扩展到 HBase、MySQL 等其他数据源
- **REQ-DATA-003**：支持实时数据查询与历史数据查询
- **REQ-DATA-004**：支持数据格式化处理（包括汇总、单位换算、字段重命名等）
- **REQ-DATA-005**：支持数据脱敏处理，严格遵守 `api_service` 的权限与资源范围控制

#### 3.1.4 可视化展示
- **REQ-VISUAL-001**：支持图表类型自动选择
- **REQ-VISUAL-002**：支持折线图展示趋势数据
- **REQ-VISUAL-003**：支持柱状图展示对比数据
- **REQ-VISUAL-004**：支持饼图展示比例数据
- **REQ-VISUAL-005**：支持表格展示明细数据

#### 3.1.5 AI助手集成
- **REQ-INTEGRATION-001**：集成到现有 AI 助手界面（云枢 Web 为首批接入方之一）
- **REQ-INTEGRATION-002**：支持对话历史管理
- **REQ-INTEGRATION-003**：支持上下文理解
- **REQ-INTEGRATION-004**：支持多轮对话
- **REQ-INTEGRATION-005**：支持移动端适配
- **REQ-INTEGRATION-006**：对外暴露统一响应结构，默认以“主图表 + 摘要文本 + 可选 widgets（如 SOP 卡片）”的形式输出，文本部分为 Markdown 格式，符合 `openspec/specs/ai-interaction.md` 中对 Markdown 与 Widget 的要求

### 3.2 非功能性需求

#### 3.2.1 性能需求
- **REQ-PERF-001**：自然语言理解响应时间 < 2秒
- **REQ-PERF-002**：SQL生成时间 < 1秒
- **REQ-PERF-003**：数据查询响应时间 < 3秒
- **REQ-PERF-004**：系统支持100+并发用户
- **REQ-PERF-005**：系统可用性 >= 99.5%

#### 3.2.2 准确性需求
- **REQ-ACC-001**：NL2SQL转换准确率 >= 80%
- **REQ-ACC-002**：查询结果准确率 >= 99%
- **REQ-ACC-003**：业务术语识别准确率 >= 90%
- **REQ-ACC-004**：时间维度理解准确率 >= 85%
- **REQ-ACC-005**：图表类型推荐准确率 >= 80%

#### 3.2.3 安全需求
- **REQ-SEC-001**：支持数据访问权限控制（复用 `api_service` 的角色 + 资源范围模型）
- **REQ-SEC-002**：支持 SQL 注入防护和查询白名单机制
- **REQ-SEC-003**：支持敏感数据脱敏
- **REQ-SEC-004**：支持查询审计日志（记录自然语言、生成的 SQL、执行耗时、结果规模等）
- **REQ-SEC-005**：支持会话安全控制
- **REQ-SEC-006**：大模型调用仅通过内部私有化部署的 OpenAI 协议兼容网关进行，禁止将真实业务数据（指标值、设备标识等）发送至模型，仅传递语义/元数据
- **REQ-SEC-007**：NL2SQL 生成与 SQL 执行必须在内网完成，不得依赖外部云端服务

## 4. 业务场景

### 4.1 数据分析场景
业务人员通过自然语言查询特定时间段的温度趋势数据。

### 4.2 异常监控场景
业务人员查询异常数据，如"华东区域温度超过28度的设备"。

### 4.3 报表生成场景
业务人员生成指定周期的能耗统计报表。

### 4.4 对比分析场景
业务人员对比不同区域的设备运行状态。

### 5.3 告警数据

## 6. 功能需求详述

### 6.1 自然语言理解功能
- 支持中文分词和词性标注
- 支持命名实体识别（时间、地点、设备等）
- 支持意图识别和槽位提取
- 支持上下文理解和多轮对话

### 6.2 NL2SQL转换功能
- 语义解析：将自然语言解析为结构化查询
- 表名映射：将业务概念映射到数据库表
- 字段映射：将业务概念映射到数据库字段
- 条件构建：构建WHERE条件子句
- 聚合操作：构建GROUP BY和聚合函数

### 6.3 查询优化功能
- SQL语法校验
- 查询性能优化
- 结果缓存机制
- 错误处理和重试

### 6.5 意图类型与处理路径

- 支持按意图对请求进行分类，并路由到不同处理分支（Agent）：
  - **数据分析类意图（ChatBI）**：如“看一下华东最近一周温度趋势”、“统计昨晚告警数量”等；
    - 处理路径：NLU → NL2SQL/LogicalQuery → 调用 Yunshu Data API Platform（api_service） → 图表 + 摘要返回；
  - **知识问答类意图（SOP / 文档 / 规范）**：如“空调漏水 SOP 是什么”、“PUE 超标怎么处理”；
    - 处理路径：意图路由到 RAG/知识库 Agent，检索内部知识库，返回 Markdown 文本 + 可选 SOP Widget；
  - **操作 / 流程类意图（Workflow）**：如“发起一次华东机房巡检”、“创建一个告警处理任务”；
    - 处理路径：意图路由到 Workflow Agent，由其调用云枢・业务服务（App Service）的 API，**MVP 阶段仅生成建议和待执行操作，不直接执行高风险动作**；
  - **说明 / 咨询类意图（Explain）**：如“怎么理解这个 PUE 指标”、“这个图代表什么含义”；
    - 处理路径：调用大模型生成解释说明，不触发实际查询或动作。
- 每类意图的处理结果均需统一包装为：主图表（如有）+ Markdown 摘要 + 可选 widgets，方便前端统一渲染。
### 6.4 可视化功能
- 图表类型自动推荐
- 数据格式化处理
- 交互式图表展示
- 图表导出功能

### 7.1 输入方式
- 文本输入：支持自然语言查询
- 语音输入：支持语音转文字查询
- 快捷查询：支持常用查询模板

### 7.2 输出方式
- 文本回复：解释查询结果
- 图表展示：可视化数据展示
- 表格展示：明细数据展示
- 下载功能：支持数据导出

### 7.3 交互流程
1. 用户输入自然语言查询
2. 系统理解用户意图
3. 生成SQL查询语句
4. 执行数据查询
5. 处理查询结果
6. 生成可视化图表
7. 返回结果给用户
