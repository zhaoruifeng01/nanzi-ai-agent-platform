# 提案：实现多智能体并行执行 (Multi-Agent Parallel Execution)

## 背景
目前平台的路由逻辑 (`RouterService`) 和编排逻辑 (`AgentService`) 采用的是 "单一路由 -> 单一智能体" 的模式。即使用户的问题跨越了多个领域（例如：“检查数据库 X 并在知识库中查找相关策略 Y”），系统也强制选择一个“最佳”智能体，这往往导致回答不完整。

## 目标
使系统支持 **多智能体并行执行** (类似 Map-Reduce 模式)。
1.  **路由 (Router)**: 能够针对单个查询选择多个智能体。
2.  **编排 (Orchestrator)**: 并行执行选定的智能体。
3.  **聚合 (Synthesis)**: 将所有智能体的结果聚合成一个统一的回复。

## 范围
- **仅后端**: 专注于 `RouterService` 和 `AgentService` 的修改。
- **前端**: 复用现有的日志流式传输；无需进行主要的 UI 重构（日志将交错显示）。

## 策略
1.  **路由升级**: 修改 Router LLM 的提示词，支持返回智能体列表（主 + 辅）或多智能体执行标志。
2.  **并行执行**: 在 `AgentService` 中，如果选择了多个智能体，则实例化多个 `Executor` 并通过 `asyncio.gather` 运行它们。
3.  **统一日志**: 将所有执行器的日志流式传输给用户（为清晰起见，前缀加上智能体名称）。
4.  **最终聚合**: 引入一个新的 `synthesis` 步骤，该步骤获取所有智能体的输出并生成连贯的最终答案。

## 关键变更
- `app/services/ai/router_service.py`: 提示词工程及针对多智能体列表的 JSON 解析。
- `app/services/ai/agent_service.py`: 重构 `chat_completion_stream` 以支持分支逻辑（单智能体 vs 多智能体）。
- `app/schemas/agent.py`: 更新 `RouteResult` 数据结构。

## 风险
- **延迟**: 并行执行受限于最慢的那个智能体。
- **Token 成本**: 运行 N 个智能体会使成本倍增。（缓解措施：路由应偏向保守）。
- **上下文窗口**: 聚合 N 个智能体的输出可能会超出聚合模型的上下文限制。（缓解措施：截断单个智能体的输出）。
