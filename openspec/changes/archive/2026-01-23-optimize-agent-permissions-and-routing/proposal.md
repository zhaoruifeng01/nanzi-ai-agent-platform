# Proposal: 优化智能体权限与路由策略

## 1. 背景 (Context)
当前的智能体路由逻辑 (`Orchestration`) 主要基于语义匹配，将用户请求分发给最合适的 Agent。
同时，权限系统 (`Permissions`) 已经支持了“用户-智能体”的绑定关系，但在执行层（对话接口）和路由层的结合尚不紧密。
此外，系统还缺乏“专家模式” (Expert Mode) 的明确支持，无法让用户强制指定与某个智能体对话。

## 2. 问题 (Problem)
1.  **路由候选集未隔离**：自动路由可能将用户无权访问的 Agent 纳入候选，导致调用失败。
2.  **权限校验不完整**：缺乏对“系统级”与“用户级”智能体的差异化权限控制。
3.  **用户控制力不足**：用户无法手动强制指定对话智能体（专家模式）。

## 3. 目标 (Goals)
1.  **权限分配优化**：管理员仅能分配“已启用”且“系统级”的智能体。
2.  **访问控制增强**：
    *   **系统级智能体**：必须拥有显式权限。
    *   **用户级智能体**：创建者拥有天然权限（前提是未禁用）。
3.  **路由逻辑收敛**：自动路由仅在“系统级且已启用”的智能体中选择。
4.  **交互优化**：
    *   **@提及**：列表仅包含有权访问的系统智能体 + 自己创建的非禁用智能体。
    *   **专家模式**：支持手动切换模式，强制指定对话对象，并提供明确的 UI 提示。

## 4. 详细需求 (Requirements)

### 4.1 权限分配 (Permission Assignment)
- **规则**：在为角色或用户分配智能体权限时，候选列表 **仅包含** `is_system=True` (系统级) **且** `is_enabled=True` (已启用) 的智能体。
- **原因**：非系统级智能体通常属于私有或特定共享范围，不通过管理员统一分配；禁用的智能体不应被分配。

### 4.2 访问鉴权 (Access Control)
- **规则**：
    1.  **系统级 (System Level)**: 用户必须在 `ai_agent_user_permissions` (或角色关联) 中拥有该智能体的显式权限。
    2.  **非系统级 (User Level)**: 如果是用户自己创建 (`created_by == current_user`) 且 **未禁用** (`is_enabled=True`)，则允许访问。

### 4.3 自动路由 (Auto-Routing)
- **规则**：`RouterService` 在自动模式下的候选集 **仅包含** `is_system=True` **且** `is_enabled=True` 的智能体。
- **原因**：确保自动路由的稳定性，只导向官方维护的 System Agents。非系统级智能体（通常是实验性或个性化的）不参与自动路由。

### 4.4 @提及列表 (Mention List)
- **规则**：用户在输入框输入 `@` 时，弹出的列表为以下两者的 **并集**：
    1.  有显式权限的系统级智能体 (`is_system=True` + Perm)。
    2.  自己创建且未禁用的非系统级智能体 (`is_system=False` + `created_by=Me` + `is_enabled=True`).

### 4.5 路由模式 (Routing Modes)
- **设置**：在设置菜单中增加“路由模式”选项。
    - **自动路由 (Auto)**: 默认模式。
    - **专家模式 (Expert)**: 用户需指定一个默认的专家智能体（范围同 @列表）。
- **专家模式行为**：
    - **UI**：聊天输入框上方显示醒目提示条：“当前为专家模式: [智能体名称]”。
    - **逻辑**：所有消息直接发送给该指定智能体，**跳过** Router 分类。
- **自动模式行为**：
    - **UI**：隐藏提示条。
    - **逻辑**：走标准 Router 流程。

## 5. 待讨论点 (Open Questions)
- 专家模式下，是否允许临时 `@` 其他智能体？（建议：允许，`@` 优先级最高）。
