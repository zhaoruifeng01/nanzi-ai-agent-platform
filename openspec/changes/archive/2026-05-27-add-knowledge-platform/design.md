## Context

云枢智能体平台已经接入 RAGFlow，用于元数据同步、知识库列表选择、`search_knowledge_base` 工具检索以及 RAGFlow 托管智能体调用。当前知识库的生命周期管理仍依赖 RAGFlow 控制台，平台内只提供“选择已有知识库”的能力，无法完成创建、上传文档、解析、删除和检索效果验证。

本变更新增「知识库开发平台」作为独立菜单分组，面向平台管理员、知识库维护人员和智能体配置人员，提供围绕 RAGFlow Dataset/Document/Retrieval API 的管理与测试能力。RAGFlow 继续承载向量知识库和文档解析检索能力，云枢平台新增本地知识库元数据表，用于维护 RAGFlow 元数据之外的业务属性、归属信息、展示信息、扩展配置和审计关联。

## Goals / Non-Goals

**Goals:**

- 在左侧导航中新增「知识库开发平台」分组，并提供「知识库管理」「检索测试」两个模块。
- 在「知识库管理」中支持知识库列表、创建、删除、文档列表、文档上传、文档删除和触发解析。
- 在「检索测试」中支持选择一个或多个知识库，配置检索参数并展示命中 chunk。
- 通过后端 Portal API 统一代理 RAGFlow OpenAPI，避免前端直接持有 RAGFlow 地址和 API Key。
- 通过本地知识库元数据表维护 RAGFlow Dataset 不覆盖的业务信息。
- 将菜单和操作纳入现有角色/权限配置体系。
- 对知识库创建、删除、文档上传、文档删除、文档解析和检索测试记录审计日志。

**Non-Goals:**

- 不在第一阶段实现审批流、知识库版本管理或文档全文预览。
- 不替代 RAGFlow 的高级解析配置、团队空间、权限模型和 Agent 编排能力。
- 不改造现有 `search_knowledge_base` 工具、RAGFlow 托管 Agent 执行链路或元数据同步流程。
- 不实现知识库内容的跨 RAGFlow 实例迁移。

## Decisions

### 决策 1：RAGFlow 作为检索事实源，平台维护本地扩展元数据

采用后端 Portal API 代理 RAGFlow datasets、documents、chunks、retrieval 接口。前端只调用云枢平台接口，不直接访问 RAGFlow。平台新增本地知识库元数据表，记录 `ragflow_dataset_id`、名称/描述快照、业务归属、可见性、标签、备注、创建人、更新人、扩展 JSON、状态等信息。

原因：
- 现有系统已经通过 `ragflow_api_url` 与 `ragflow_api_key` 集中配置 RAGFlow。
- 避免前端暴露 RAGFlow API Key。
- RAGFlow 原生 Dataset 元数据不足以承载平台侧业务管理信息。
- 本地表可以支撑后续权限、审计、运营标注和展示优化。

备选方案：
- 仅做 RAGFlow 薄代理，不建本地表。该方案实现最快，但无法维护更多业务元数据，后续扩展会受限。
- 完整双向同步 RAGFlow 所有 Dataset/Document 字段。该方案信息最全，但同步复杂度高；第一阶段只保存平台需要的扩展元数据和 RAGFlow ID 映射。

### 决策 2：知识库管理和检索测试拆成两个页面

「知识库管理」负责资源生命周期，「检索测试」负责验证检索效果和调参。

原因：
- 两类任务使用频率、布局和用户心智不同。
- 管理页强调列表、操作和状态；测试页强调 query、参数和结果解释。
- 拆分后更容易做权限控制，例如只允许部分角色测试检索但不允许删除知识库。

备选方案：
- 在一个页面通过 Tab 合并管理与测试。该方案初期少一个路由，但页面会过重，也不利于菜单级权限控制。

### 决策 3：复用现有 RAGFlow 客户端并补齐 Portal API

后端优先复用 `app/services/ai/ragflow_client.py` 已存在的 `list_datasets`、`create_dataset`、`delete_datasets`、`list_documents`、`upload_document`、`delete_documents`、`parse_documents` 和 `retrieve` 方法。Portal endpoint 负责参数校验、权限校验、异常翻译和响应结构统一。

原因：
- RAGFlow API 调用逻辑已集中在 `RagFlowClient`，继续扩展可减少重复。
- Portal endpoint 更适合承载用户权限和 HTTP 语义。

备选方案：
- 在 endpoint 内直接写 httpx 请求。该方案短期快，但会复制已有客户端逻辑，后续维护成本更高。

### 决策 4：权限采用菜单权限 + 操作权限

新增 `menu:knowledge_management`、`menu:knowledge_retrieval_test` 等菜单权限，以及创建、删除、上传、解析、测试等 element 权限。前端用于展示/隐藏入口，后端用于强制校验。

原因：
- 符合现有 `MENU_TREE`、`v-has-perm` 和后端权限服务模式。
- 删除、上传、解析等操作具有破坏性或资源成本，不能只依赖前端隐藏。

备选方案：
- 仅管理员可用。该方案最简单，但无法支持知识库维护人员这类非管理员角色。

### 决策 5：上传不限制格式，解析手动触发

平台侧不限制上传文件类型，文件能否解析由 RAGFlow 返回结果决定。上传成功后不自动解析，用户需要手动点击解析。

原因：
- RAGFlow 支持的格式和解析能力可能随版本变化，平台侧不应过早固化白名单。
- 手动解析可避免批量上传或大文件上传后立即触发高成本解析任务。
- 用户可先确认文档列表和选择范围，再主动解析。

备选方案：
- 平台限制常见格式并自动解析。该方案体验更短，但不利于兼容 RAGFlow 新格式，也更容易触发不可控解析任务。

### 决策 6：检索测试直接调用 `/api/v1/retrieval`

检索测试页通过后端代理调用 `RagFlowClient.retrieve()`，而不是调用 RAGFlow Agent 的 OpenAI 兼容 chat 接口。

原因：
- 检索测试关注 chunk 命中质量，不关注最终生成答案。
- `/api/v1/retrieval` 支持明确传入 `dataset_ids`、`top_k`、`similarity_threshold`、`vector_similarity_weight` 等参数。
- 测试结果可直接用于调试 `search_knowledge_base` 工具效果。

备选方案：
- 调用 RAGFlow Agent chat 接口。该方案接近最终问答体验，但 dataset 由 RAGFlow App 绑定，难以作为通用知识库检索调试工具。

### 决策 7：所有知识库操作记录审计日志

知识库创建、删除、文档上传、文档删除、文档解析、检索测试均记录审计日志。日志至少包含操作者、操作类型、目标 dataset/document、请求摘要、结果状态、错误信息和 trace/request 标识。

原因：
- 知识库内容会影响智能体回答质量，需要可追溯。
- 删除和解析属于高影响操作，必须可审计。
- 检索测试可能暴露知识库内容命中片段，也需要记录谁在何时测试了什么范围。

备选方案：
- 仅依赖通用访问日志。该方案粒度不足，不利于按知识库对象追踪操作历史。

## Risks / Trade-offs

- RAGFlow API 返回结构在不同版本可能不一致 → 后端响应解析需要兼容 `data` 为 list、dict、`datasets`、`docs` 等结构，并保留原始错误信息用于排查。
- 文档上传和解析可能耗时较长 → 上传接口设置合理超时，解析操作返回触发结果，解析状态通过文档列表刷新查看。
- 删除知识库/文档是破坏性操作 → 前端必须使用自定义确认弹窗，后端必须校验操作权限。
- 本地元数据与 RAGFlow Dataset 可能不一致 → 创建、删除等写操作采用“先 RAGFlow 成功，再写/更新本地表”的顺序，并在页面中对失联或缺失的 RAGFlow Dataset 显示异常状态。
- 非管理员数据可见性存在边界 → 沿用现有用户资源权限过滤规则；管理类操作要求对应 element 权限，有权限的用户可以创建、删除、上传和解析。
- RAGFlow 服务不可用会影响页面 → 页面必须展示连接失败、配置缺失和重试入口，避免空白页。
- 审计日志可能记录敏感请求内容 → 检索测试日志记录 query 摘要和 dataset 范围，遵循现有敏感字段脱敏规则，不记录 API Key。

## Migration Plan

1. 新增权限常量和路由，默认管理员可见；普通角色需在角色管理中显式勾选。
2. 新增知识库本地元数据表数据库迁移，字段允许扩展 JSON 以承载后续业务属性。
3. 扩展后端 RAGFlow Portal API，保持原有 `/datasets` 列表接口兼容。
4. 新增前端页面并接入菜单。
5. 部署后由管理员验证 RAGFlow 配置、知识库列表、创建、上传、解析、删除、审计日志和检索测试流程。
6. 如需回滚，移除新增菜单入口即可停止用户访问；后端新增接口不影响既有调用，本地元数据表可保留。
