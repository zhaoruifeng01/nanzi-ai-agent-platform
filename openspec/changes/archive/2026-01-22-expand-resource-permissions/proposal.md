# Proposal: 扩展资源权限体系以支持菜单与功能点控制

## 1. 动机 (Why)
当前系统的权限模型主要管控后端资源（智能体、数据集、API），前端界面的控制主要依赖于系统级角色（`admin`/`user`）的粗粒度硬编码判断。随着系统功能的复杂化，需要支持：
- **定制化导航**：不同职能的用户登录后应看到不同的功能菜单。
- **操作保护**：特定的敏感操作（如“删除数据集”、“修改系统配置”）应仅对获得授权的人员开放，即便他们同属普通用户组。
- **UI 一致性**：通过统一的权限模型，实现前端 UI 元素与后端执行权限的同步。

## 2. 核心变更 (How)
### 后端变更：
- **数据模型优化**：在权限校验逻辑中支持新的资源类型标识：`menu` (菜单) 和 `element` (按钮/功能点)。
- **权限服务升级**：更新 `PermissionService`，在聚合权限时包含菜单和功能点，并优化返回结构，方便前端一次性加载。
- **元数据同步**：在权限管理界面支持对菜单项和按钮标识的配置。

### 前端变更：
- **动态菜单渲染**：重构左侧导航栏逻辑，根据后端返回的 `menu` 权限列表进行过滤。
- **权限指令开发**：引入 Vue 指令（如 `v-perm="'user:delete'"`），实现按钮级的显隐控制。
- **路由守卫增强**：在全局路由守卫中增加对 `menu` 权限的校验，防止通过 URL 直接越权访问。

## 3. 预期效果 (Rationale)
- **更强的安全性**：实现了从 UI 到 API 的全链路权限闭环。
- **更好的用户体验**：用户界面更简洁，仅展示其有权操作的功能。
- **高可扩展性**：新增菜单或按钮时，只需在权限表中注册对应的 ID 即可，无需修改前端硬编码。
