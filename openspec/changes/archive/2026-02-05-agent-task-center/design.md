## Context

云枢智能体平台目前是一个基于请求-响应模式的交互系统。为了满足企业内部对自动化报表、主动监控和定期巡检的需求，需要引入一个能够自主触发执行的“智能体任务中心”。

## Goals / Non-Goals

**Goals:**
- **可靠调度**: 确保任务在指定的时间准时触发。
- **分布式安全**: 在多节点部署环境下，确保同一任务在同一触发点仅由一台服务器执行。
- **权限闭环**: 能够在后台执行时完整恢复任务创建者的权限上下文，通过数据和工具的权限校验。
- **动态 Prompt**: 支持在指令中注入动态时间变量（如昨日、上月）。
- **管理透明**: 提供可视化界面查看任务队列、下一次运行时间及执行历史。

**Non-Goals:**
- **复杂流编排**: 本任务中心不负责智能体内部的节点流转逻辑（那是 RAGFlow 或 LangGraph 的职责）。
- **秒级高频调度**: 目标场景为分钟级/小时级/日级的业务任务。

## Decisions

### 1. 调度引擎：APScheduler + SQLAlchemy JobStore
- **选择**: 使用 Python `APScheduler` 库。
- **理由**: 与现有 FastAPI 项目集成简单，支持 Cron、Interval 等多种触发模式。通过配置 `SQLAlchemyJobStore` 直接将任务定义序列化存储在 MySQL 中，确保服务重启后任务自动恢复。

### 2. 分布式并发控制：Redis 分布式锁
- **策略**: 在任务触发时，各节点尝试在 Redis 中抢占 `lock:task:{task_id}:{timestamp}`。
- **理由**: `APScheduler` 虽然支持数据库存储，但在多节点同时扫描数据库时可能存在微妙的竞态条件。引入 Redis `SET NX` 锁作为第二层保障，实现简单且性能极高。

### 3. 身份模拟 (Impersonation) 与安全性
- **方案**: 在 `agent_scheduled_tasks` 表中关联 `user_id`。
- **逻辑**: 任务中心执行时，通过 `user_id` 从数据库加载该用户的 API Key 或加密凭证，构造一个模拟的 `user_info` 上下文注入到 `agent_service`。
- **安全性**: 所有定时任务产生的 SQL 查询仍会经过系统现有的 `SQL 防火墙` 和 `权限过滤器`。

### 4. 变量预处理
- **方案**: 内置 `TemplateProcessor`。
- **语法**: 使用 `{{today-1d}}` 或 `{{yesterday}}`。
- **逻辑**: 执行引擎在发送 Prompt 给 LLM 前，先进行正则匹配和时间偏移计算，完成静态替换。

## Risks / Trade-offs

- **[Risk] 用户凭证失效** → **[Mitigation]** 若用户离职或 API Key 被禁用，任务执行将报错并记录到日志中，管理员可在 UI 中收到提醒。
- **[Risk] AI 回复不确定性** → **[Mitigation]** 定时任务执行时，若 AI 未能生成预期格式的结果，系统支持配置“失败重试”策略。
- **[Trade-off] 性能压力** → 定时任务在大批量触发时（如所有任务都设在 8:00）可能导致数据库压力瞬间升高。通过在调度时加入微小的随机抖动（Jitter）来平滑峰值。
