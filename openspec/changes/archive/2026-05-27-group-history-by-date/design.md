## 架构设计背景 (Context)

云枢智能体平台嵌入式聊天侧边栏目前完全是扁平化展示所有历史会话（按 Trace 级别或排重后的会话）。在面对长期的智能体 Prompt 调试时，长列表的阅读难度极高。引入相对日期分组（今天、昨天、3天前、7天前、更早）将有助于整理会话。

为了进一步优化开发调试的闭环体验，除了将会话按日期聚合，还需提供在时间分组维度下一键批量删除该组内全部会话的能力，使用户可以快速清理某一天或某一周产生的冗余垃圾测试数据。这涉及前端展示组件重构、事件交互流，以及后端全新批量级联删除接口的设计。

## 目标与非目标 (Goals / Non-Goals)

### 目标 (Goals)
1. **时间分组显示**：在 `ChatHistorySidebar.vue` 侧边栏中，将会话历史以“今天”、“昨天”、“3天前”、“7天前”、“更早”五个标准日期组呈现。
2. **批量删除 API**：提供一个高效的批量删除接口，接收一个包含多个会话 ID 的数组，在数据库的单个事务中完成级联物理或逻辑删除。
3. **分组一键删除交互**：在分组 Header 上点击一键删除后，弹出确认框，前端向后端批量删除接口发起调用，完成后彻底移除前端对应分组卡片。
4. **当前会话的自动切换**：若一键删除包含了用户当前正在聊天的会话，删除完成后必须清空视窗，自动初始化生成一个全新的空白会话。

### 非目标 (Non-Goals)
1. 不涉及对现有 `GET /api/v1/chat/history` 查询接口响应结构的修改（分组算法完全由前端响应式计算属性 `groupedHistoryList` 实时计算完成，以维护后端接口的通用性与向下兼容）。
2. 不支持跨分组的任意多选删除。只支持特定日期分组内的一键清空或现有的单条 Trace/会话删除。

## 技术方案决定 (Decisions)

### 决定 1: 前端实时分组 vs 后端直接返回分组结构
- **选择**：采用 **前端实时分组**。通过父组件 `EmbedChat.vue` 的 `computed` 计算属性 `groupedHistoryList` 来完成对原始列表的分发，子组件 `ChatHistorySidebar.vue` 接受此聚合对象数组做嵌套遍历渲染。
- **对比与论证**：
  - *方案 A (后端分组返回)*：需要修改后端 `/history` API 的结构，或增加分组逻辑。这不仅会破坏其他使用该接口的模块，也会在进行前端搜索、过滤或分页滚动（Load More）追加数据时，加大后端重新划分时间组、维护状态的复杂度。
  - *方案 B (前端实时分组)*：后端依然只提供扁平的列表（带分页和关键字过滤），前端收到响应后在内存中通过相对时间戳动态分拣。这种方案逻辑独立，在用户输入过滤关键字或向下滚动分页拉取更多历史时，前端计算属性能自动且瞬间重新计算并流畅渲染，维护成本极低。

### 决定 2: 相对时间区间的基准划分
- **选择**：以 **“今天凌晨 00:00:00” 作为基准原点** 向前计算偏移。
- **分拣算法边界条件**：
  - 基准点：`startOfToday = 今天的 00:00:00`
  - **今天**：创建时间 $\ge startOfToday$。
  - **昨天**：$startOfToday - 1 \text{天} \le$ 创建时间 $< startOfToday$。
  - **3天前**：$startOfToday - 3 \text{天} \le$ 创建时间 $< startOfToday - 1 \text{天}$。
  - **7天前**：$startOfToday - 7 \text{天} \le$ 创建时间 $< startOfToday - 3 \text{天}$。
  - **更早**：创建时间 $< startOfToday - 7 \text{天}$。
- **论证**：基于日历天（00:00）的分组比单纯基于“当前毫秒数减去 24 小时毫秒数”的分组方式更符合直觉。

### 决定 3: 后端批量级联删除实现 (POST /api/v1/chat/history/batch-delete)
- **选择**：在后端新增一个路由 `POST /api/v1/chat/history/batch-delete`，在单个数据库事务（Transaction）中，对于所有传入的 `conversation_ids` 进行级联删除。
- **数据清理范围**：
  - 删除 `chat_history`（或对应聊天消息记录表）中 `conversation_id` 匹配的所有记录。
  - 删除 `session_trace`（智能体运行 Trace 日志表）中对应的 trace 记录。
  - 级联删除 Trace 步骤（Steps）、相关关联记录。
- **论证**：这样可以在单次数据库连接中将相关数据全部清除，防止数据库孤立垃圾数据的产生，且单次批量请求开销远低于多次单条 `DELETE` 请求。

## 潜在风险与折衷方案 (Risks / Trade-offs)

### [风险] 批量删除大量数据导致数据库死锁或网络超时
- **影响**：如果一个日期分组下有数万条历史测试消息，后端在一个事务内做超大规模的级联物理删除，可能导致数据库行锁冲突或响应超时。
- **缓解措施**：
  1. 侧边栏本身做分页加载（每页 20 条），一个分组内加载出来的可删除项最多几十条（除非用户疯狂滚动到底部），单次批量删除的数据规模一般非常小，在毫秒级即可完成。
  2. 后端执行批量删除时，在 SQL 语句或 SQLAlchemy ORM 中，对 `conversation_id` 施加带索引的批量删除命令，避免全表扫描以保证执行效率。
