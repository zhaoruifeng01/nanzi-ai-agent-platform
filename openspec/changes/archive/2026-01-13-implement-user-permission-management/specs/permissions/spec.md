## ADDED Requirements

### Requirement: 用户资源权限管理
> 系统必须 (MUST) 支持为特定用户分配对智能体、数据集和工具的访问权限，并支持基于角色的架构扩展。

#### Scenario: 管理员为用户分配资源权限
> 管理员在用户管理界面为特定用户分配智能体、数据集和工具的访问权限。

- **Given** 管理员已登录并处于“用户管理”界面。
- **When** 点击某用户的“编辑”或“权限设置”按钮。
- **Then** 界面展示“权限配置”选项卡，包含“智能体”、“数据集”、“接口”三个子标签。
- **When** 在“智能体”标签页中勾选若干个智能体并保存。
- **Then** 后端更新该用户的权限记录。
- **And** 该用户的权限缓存被清除/更新。

### Requirement: 资源访问控制 (Enforcement)
> 系统必须 (MUST) 在业务执行层面强制校验用户对资源的访问权限。

#### Scenario: 获取数据集列表时的权限过滤
- **Given** 普通用户 A 仅拥有数据集 D1 的权限。
- **And** 系统中存在数据集 D1, D2, D3。
- **When** 用户 A 请求获取数据集列表 (List Datasets)。
- **Then** 返回结果中仅包含 D1。
- **And** D2, D3 不可见。

#### Scenario: 智能体对话时的权限校验
- **Given** 普通用户 B 没有智能体 Agent-X 的权限。
- **When** 用户 B 尝试向 Agent-X 发送对话消息。
- **Then** 系统拒绝执行该请求。
- **And** 返回明确的错误提示“无权访问该智能体”。


#### Scenario: 权限缓存自动刷新
> 权限变更后，系统应确保缓存及时更新，避免权限滞后。

- **Given** 用户 A 的权限已被缓存到 Redis。
- **When** 管理员通过 API 或 界面修改了 用户 A 的权限。
- **Then** 系统自动删除 用户 A 在 Redis 中的权限缓存 key。
- **When** 用户 A 下次访问受控资源或刷新页面。
- **Then** 系统从数据库重新加载权限并写入 Redis。

#### Scenario: 数据库结构支持 RBAC 扩展
> 数据库设计应预留角色表，即使当前仅使用用户-资源直接绑定。

- **Given** 数据库初始化完成。
- **Then** 存在 `ai_agent_roles` 表（可能为空）。
- **And** 存在 `ai_agent_resource_permissions` 表，且包含 `role_id` 字段（允许为空）。
