# 个人业务工作台设计

**日期：** 2026-07-18  
**状态：** 已批准，待实施计划

## 1. 背景

平台已经具备通知、智能体任务、黄金报表、订阅简报、ChatBI 会话、授权智能体和场景模板等能力，但普通业务用户仍需进入不同模块寻找待处理事项、最新结果和最近工作。现有导航主要按智能体、ChatBI、知识库和系统管理等技术模块组织，缺少面向个人日常工作的统一入口。

个人业务工作台不创建新的任务、通知或报表体系，而是在权限边界内聚合现有业务数据，并为每条内容提供返回原业务模块的明确动作。

## 2. 目标

1. 首屏优先回答“今天需要处理什么”“最近产生了什么结果”“可以从哪里继续工作”。
2. 同时服务普通业务人员和部门负责人，首版以普通业务人员为默认视图。
3. 根据用户真实数据动态呈现活跃、有事项、平静日和新用户状态。
4. 没有内容时提供有效引导，不展示虚构指标或一排全零卡片。
5. 复用现有业务能力和权限，不在工作台复制任务、报表、会话等模块的编辑功能。

## 3. 非目标

首期不实现：

- 团队经营驾驶舱或部门级指标聚合。
- AI 自动生成待办或推断不存在的运行状态。
- 工作区文件和生成文件的统一产出索引。
- 用户自由拖拽、增删组件的工作台配置器。
- 管理员配置任意工作台组件。
- 新的通知、任务、报表或会话业务模型。

## 4. 用户与入口

### 4.1 用户类型

- 普通业务人员：查看本人待处理事项、最新结果、最近工作和常用助手。
- 部门负责人：首版使用相同个人视图；团队关注区域留到后续版本。
- 管理员：保留现有平台概览页，不强制用业务工作台替换管理看板。

### 4.2 页面入口

新增路由 `/dashboard/workbench`，页面名称“我的工作台”。

- 普通业务用户登录后默认进入工作台。
- 管理员仍可进入现有 `/dashboard` 概览页。
- 移动端底部导航将工作台作为首页。
- 智能助手、数据首页、任务中心、个人中心和管理模块入口继续保留。
- 路由使用现有业务用户可用权限，不在首期新增数据库菜单权限；具体复用的权限键在实施计划中根据当前种子数据确认。

## 5. 信息架构

首屏采用“行动优先型”结构，优先级固定为：

1. 待处理事项。
2. 最新结果。
3. 继续工作。
4. 常用助手或推荐场景。

页面组件边界：

- `WorkbenchAttention`：失败任务、待确认事项和未读结果。
- `WorkbenchResults`：智能简报、黄金报表结果等最新产出。
- `WorkbenchResume`：最近智能体会话、ChatBI 分析和报表活动。
- `WorkbenchAgents`：常用授权智能体。
- `WorkbenchScenarios`：新用户或低活跃用户可用的场景模板。
- `WorkbenchHealthyState`：没有待处理事项时的正常状态。
- `WorkbenchOnboarding`：没有历史行为时的首次使用引导。

各组件只负责展示和导航，不承载原业务对象的编辑表单。

## 6. 动态状态

### 6.1 活跃用户且有事项

展示顺序为待处理、最新结果、继续工作、常用助手。待处理区域始终置顶，每类最多显示有限条目并提供“查看全部”。

### 6.2 活跃用户但无事项

使用轻量状态条展示“今日运行正常，暂无失败任务和待确认事项”，不保留空白待办卡片。继续工作成为首要操作，其后展示最新结果和常用助手。

### 6.3 新用户或无历史用户

不展示全零统计。页面提供用户有权限访问的业务场景、授权智能体和推荐问题，优先引导完成第一次有效工作。

场景推荐来自平台真实场景模板，并根据其资源和权限可用性过滤。没有可安装场景时退化为授权智能体；两者都为空时展示联系管理员开通资源的明确说明。

### 6.4 局部无内容

- 没有报表或简报：隐藏最新结果中的对应类型。
- 没有任务：不显示任务数量，提供创建定时任务的入口仅限用户具备相应权限时。
- 没有最近会话：用授权智能体或场景推荐替代继续工作区域。
- 某数据源请求失败：对应区域显示“暂时无法获取”，其他区域继续可用。

## 7. 数据来源

首期只使用现有系统产生的真实数据：

| 工作台内容 | 数据来源 |
|---|---|
| 未读结果与业务通知 | 站内通知及未读状态 |
| 定时任务、失败运行与下一次执行 | 智能体任务、任务运行日志、黄金报表订阅 |
| 智能简报与黄金报表结果 | 报表运行历史、结果快照和简报交付记录 |
| 最近工作 | 智能体执行历史、ChatBI 会话、最近报表运行 |
| 常用助手 | 当前用户有权访问的智能体及执行历史 |
| 推荐场景 | 场景模板目录及其资源可用性 |

现有 `/api/portal/data-portal/home` 已聚合报表、简报和 ChatBI 会话。个人工作台聚合层可复用其查询和纯函数构造逻辑，但不得让工作台服务反向依赖前端数据门户组件。

## 8. 聚合接口

新增：

```text
GET /api/portal/workbench/home
```

建议响应结构：

```json
{
  "mode": "active",
  "attention": [],
  "latest_results": [],
  "resume_items": [],
  "favorite_agents": [],
  "recommended_scenarios": [],
  "next_scheduled_item": null,
  "source_status": {
    "notifications": "ok",
    "tasks": "ok",
    "reports": "ok",
    "conversations": "ok",
    "agents": "ok",
    "scenarios": "ok"
  }
}
```

`mode` 取值：

- `active`：存在待处理事项。
- `quiet`：存在历史行为但无待处理事项。
- `new_user`：不存在可恢复工作、近期结果或历史行为。

聚合接口负责权限过滤、条目排序、去重、数量限制和局部错误隔离。前端不得自行请求多个接口后推导用户状态。

## 9. 条目契约与动作

各列表使用统一的轻量条目协议：

```json
{
  "id": "stable-id",
  "type": "task_failure",
  "title": "库存巡检执行失败",
  "subtitle": "今天 09:12",
  "occurred_at": "2026-07-18T09:12:00+08:00",
  "status": "failed",
  "action": "open_task_log",
  "target": {
    "task_id": 17,
    "run_id": 302
  }
}
```

动作映射：

| 内容 | 动作 |
|---|---|
| 失败任务 | 打开任务中心并定位运行日志 |
| 智能简报 | 打开黄金报表运行详情和推送内容 |
| 待确认工具调用 | 恢复原会话并定位确认状态 |
| 最近智能体或 ChatBI 分析 | 打开原会话并恢复上下文 |
| 黄金报表活动 | 打开报表详情或指定运行记录 |
| 常用助手 | 创建或恢复该智能体会话 |
| 场景模板 | 打开模板详情或安装向导 |
| 推荐问题 | 携带问题进入智能助手 |

目标对象不存在、已删除或权限变化时，接口不返回不可达动作；若状态在点击后才变化，目标页按现有权限处理并显示友好提示。

## 10. 排序与数量限制

- 待处理：严重程度优先，其次按时间倒序，首屏最多 5 条。
- 最新结果：按完成时间倒序，首屏最多 6 条。
- 继续工作：按最近活动时间倒序并按会话或业务对象去重，最多 4 条。
- 常用助手：近期使用优先，再按平台排序；没有使用历史时按平台推荐顺序，最多 6 个。
- 推荐场景：推荐标记和资源可用性优先，最多 4 个。

同一报表运行产生运行记录、简报和通知时，最新结果只保留信息最完整的简报；通知仍可出现在待处理区域，但必须使用稳定业务键避免重复展示。

## 11. 错误处理与性能

- 每个数据源独立查询和归一化；单个来源失败不得使整个接口失败。
- `source_status` 标记各来源为 `ok`、`empty` 或 `error`，可附带稳定错误码，不能返回底层敏感异常。
- 接口设置统一时间范围和数量上限，不拉取完整历史后在 Python 中截断。
- 批量查询用户可见资源，禁止按列表项逐条查询形成 N+1。
- 聚合结果首期不做长时间 Redis 缓存，前端可做短时请求去重并保留上一次成功数据。
- 页面后台刷新时不闪回空态；刷新失败保留旧数据并提示对应区域暂时无法更新。

## 12. 移动端

- 与桌面端使用相同数据和信息优先级，不另建移动端业务逻辑。
- 待处理事项、最新结果和继续工作采用单列卡片。
- 常用助手和场景使用横向滑动或双列卡片。
- 首屏避免宽表格；报表详情和任务日志进入已有页面或抽屉。
- 点击后返回工作台时保留滚动位置和上一次成功数据。

## 13. 权限与安全

- 所有聚合查询以当前登录用户为边界。
- 报表继续遵守所有者、用户分享、角色分享和数据运行权限。
- 会话、任务、通知和智能体只返回当前用户有权查看的对象。
- 推荐场景不得因为模板存在就泄露未授权的数据集、知识库或工具名称。
- 部门负责人首版也不能查看团队成员数据，除非既有权限模型已经明确授权。
- 工作台不通过模型推断用户状态，不把模型文本当作任务或运行事实。

## 14. 测试与验收

### 14.1 后端

- 聚合接口分别覆盖 `active`、`quiet` 和 `new_user` 三种模式。
- 单个来源异常时接口仍返回其他来源数据及正确 `source_status`。
- 不同用户、角色和分享方式不会产生越权数据。
- 同一业务事件跨通知、报表运行和简报记录时正确去重。
- 被删除或权限失效的目标不返回不可达动作。
- 查询数量受控，不随列表条目线性增长。

### 14.2 前端

- 新用户不出现全零卡片。
- 无待办用户显示正常状态且可以继续最近工作。
- 有待办时待处理区域置顶。
- 局部来源错误不遮蔽整个页面。
- 每一种动作能够跳转并定位到原业务对象。
- 桌面和移动端均可完成“发现事项、打开原业务、返回工作台”的闭环。
- 刷新时保留上一次成功内容，不闪烁为空页面。

### 14.3 兼容性

- 不修改现有通知、任务、报表、会话和智能体接口契约。
- 不替换管理员概览页。
- 用户仍可通过原导航直接进入各业务模块。
- 聚合接口或工作台页面异常不影响智能助手及其他业务页面。

## 15. 实施阶段

1. 建立聚合服务、响应模型和后端契约测试。
2. 接入通知、任务、报表、会话、授权智能体和场景模板数据。
3. 实现工作台页面、三种动态状态和局部错误降级。
4. 打通任务、简报、会话、报表、智能体和场景的导航动作。
5. 增加移动端入口、返回状态恢复和前端契约测试。
