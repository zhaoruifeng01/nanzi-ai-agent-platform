# ChatBI 非查数引导与查数澄清分流设计

## 背景

数据智能体当前使用 `CLARIFICATION_OR_NON_DATA` 同时承载两类不同请求：

1. 身份、模型、能力咨询、闲聊、写作等非查数需求；
2. 已确认属于查数，但统计对象、指标、时间范围或上下文不足的需求。

两类请求共用类型和展示逻辑，导致“现在是什么模型”等非查数问题可能进入通用查数澄清模板，并出现“补充时间、补充指标口径、补充统计对象”等无关按钮。

## 目标

- 非查数需求只引导用户切换到自动路由或其他智能体，不伪装成查数需求。
- 只有能够确认存在查数意图、但执行条件不足的请求才进入查数澄清。
- 用户主动选择数据智能体后不自动切换，是否切换由用户确认。
- 保持正常查数、元数据探索、结果复用、上下文动作和技能执行链路不变。

## 非目标

- 不在本次改造中实现自动切换智能体。
- 不调整外层智能体路由策略。
- 不扩展数据查询、Schema 检索或 SQL 执行能力。
- 不尝试让数据智能体回答通用知识、模型配置或写作类问题。

## 方案选择

采用“分类类型拆分 + 结构化缺口”的方案。

不采用单纯扩充关键词的方案，因为固定表达无法覆盖持续出现的新说法。不采用只修改展示文案的方案，因为底层分类仍会混淆非查数与查数澄清，后续仍可能触发错误的按钮、日志或模型调用。

## 分类模型

将原有 `CLARIFICATION_OR_NON_DATA` 拆分为两个语义明确的类型：

- `NON_DATA_REQUEST`：已判断不是数据查询。
- `CLARIFICATION_REQUIRED`：已判断具有数据查询意图，但缺少安全执行所需的信息。

分类结果增加结构化缺口字段 `missing_fields`。建议支持以下值：

- `data_object`：统计对象或业务数据范围。
- `metric`：指标或统计口径。
- `time_range`：时间范围。
- `dimension`：分组或分析维度。
- `result_context`：需要复用的上一轮查询结果。
- `dataset_or_schema`：存在多个相近数据集、表或字段，无法安全选择。

`reasoning` 继续用于日志和诊断，不再作为展示层推断场景和按钮的主要数据源。

## 分类顺序

DataQuery 内部分类按以下优先级执行：

1. 元数据探索、格式调整、结果复用、上下文动作、技能执行等现有高置信规则。
2. 明确的新数据查询或联邦查询。
3. 高置信非查数快速规则，例如身份、模型、能力、闲聊、写作、翻译等表达。
4. 其余请求交给分类 LLM，在 `data_query`、`clarification_required`、`non_data_request` 等完整类型中选择。
5. LLM 选择 `clarification_required` 时必须同时返回至少一个 `missing_fields`；否则降级为 `non_data_request`，避免无依据地生成通用查数缺口。

规则关键词只承担低延迟快速通道，不作为完整语义覆盖方案。分类 LLM 的提示词需要明确区分：

- “现在是什么模型”属于非查数需求；
- “查一下销售情况”具有查数意图但指标或时间等条件不足；
- “查本月各区域销售额”是可继续执行的新数据查询。

## 运行时处理

### 非查数请求

`NON_DATA_REQUEST` 进入独立的非查数引导 handler：

- 不检索 Schema；
- 不执行 SQL；
- 不调用查数澄清生成 LLM；
- 简短说明当前数据智能体负责业务数据查询与分析；
- 提供“切换智能体”按钮，目标为 `/switch_to_auto`；
- 可提供“查看我能查哪些数据”按钮；
- 禁止展示查数缺口和“补充时间/指标/统计对象”按钮。

用户点击切换前保持在数据智能体，不静默改变专家模式。

### 查数澄清

`CLARIFICATION_REQUIRED` 进入查数澄清 handler：

- 根据 `missing_fields` 生成原因说明和 quick 按钮；
- 只展示实际缺少的内容；
- `result_context` 使用“先完成一次数据查询”或“明确基于哪次结果”的引导；
- `dataset_or_schema` 展示候选数据集、指标或字段供选择；
- 不再在缺口为空时默认生成“统计对象、指标口径、时间范围”三件套。

如果无法确认查数意图或无法形成结构化缺口，应回到 `NON_DATA_REQUEST`，而不是进入查数澄清。

## 展示与日志

两类响应使用不同标题：

- 非查数：`当前问题更适合其他智能体`
- 查数澄清：`需要补充查数信息`

SSE 日志和前端思考面板应反映真实类别。非查数请求不得使用“系统判断当前问题尚不足以直接执行查数”等容易暗示用户补充查数参数的文案。

## 兼容性

- 保留旧枚举值的临时兼容解析，防止历史测试、缓存数据或未同步调用方立即失败；新分类流程不再产出旧值。
- 现有 `NEW_DATA_QUERY`、`METADATA_QUERY`、`REUSE_PREVIOUS_RESULT`、`FORMAT_CORRECTION`、`CONTEXT_ACTION`、`SKILL_EXECUTION` 和 `FEDERATED_DATA_QUERY` 行为不变。
- 原有 `/switch_to_auto` 前端指令继续复用，不新增前端协议。

## 测试设计

### 分类测试

- “现在是什么模型”“你用的哪个模型”“帮我写一封邮件”“翻译一下这段话” => `NON_DATA_REQUEST`。
- “你是谁”“你能做什么”继续命中非查数快速路径，并跳过分类 LLM。
- “查一下销售额”“看看业务情况”在能够识别查数语义时 => `CLARIFICATION_REQUIRED`，并返回具体 `missing_fields`。
- “查本月各区域销售额” => `NEW_DATA_QUERY`。
- “你能查哪些字段” => `METADATA_QUERY`。
- “分析刚才结果”且无可复用结果 => `CLARIFICATION_REQUIRED`，缺口为 `result_context`。

### 响应测试

- 非查数响应包含“切换智能体”，不包含“补充时间、指标口径、统计对象”。
- 查数澄清按钮严格对应 `missing_fields`。
- 缺口为空时不得生成默认三件套。
- 非查数请求不调用 Schema、SQL 和澄清生成 LLM。
- 正常查数、元数据探索和结果复用的既有测试继续通过。

## 验收标准

1. 在数据智能体中输入“现在是什么模型”，只看到职责说明和切换智能体引导。
2. 非查数请求不出现任何查数参数补充按钮。
3. 明确查数但信息不足时仍能获得针对性的澄清选项。
4. 信息完整的查数请求不新增澄清步骤。
5. 既有 ChatBI 分类与 runner 回归测试通过。
