# 事实取证门禁柔性风险提示优化设计

## 1. 背景

现有平台级事实取证门禁采用严格的凭证类型匹配：请求被判定为某类事实来源后，候选回答必须取得完全匹配的 `EvidenceReceipt`，否则丢弃正文并发送 `grounding_blocked` 事件。该策略能够阻止无依据事实输出，但在路由判源不准确、知识库与附件来源重叠、专业智能体直选、工具已成功但凭证类型不一致等场景中产生较高误拦截率。

本设计将事实取证门禁由“阻断式门禁”调整为“柔性风险提示”。事实证据不足或类型不一致时不再丢弃模型正文、不再展示阻断卡片，而是在消息正文中加入清晰、克制的风险提示。工具权限、安全审计、SQL 安全和敏感操作确认等安全门禁不受影响，仍按原有策略硬阻断。

本设计覆盖 `2026-07-10-evidence-grounding-gate-design.md` 中关于事实取证失败时 `fail closed`、替换正文和展示阻断入口的要求；证据账本、可信工具签发和类型化凭证仍继续保留。

## 2. 目标

1. 显著降低知识库、附件、内部数据等可信来源之间因类型不一致造成的误拦截。
2. 事实取证不足时保留模型回答，通过正文风险提示向用户表达不确定性。
3. 不再发送面向用户的 `grounding_blocked` 阻断卡片，不再提供“重新核实/查看分析方法”阻断按钮。
4. 保留证据账本和审核结果，用于提示分级、日志、指标和后续质量分析。
5. 不把“任意工具调用成功”当作通用事实凭证，继续识别无关来源和失败结果。
6. 不削弱权限、SQL、安全审计、外部执行确认等真正的安全边界。

## 3. 非目标

1. 不实现逐句事实蕴含或逐数字引用校验。
2. 不重写 Router、ChatBI 或 Knowledge Agent 的主体执行架构。
3. 不取消工具证据类型注册和成功结果判断。
4. 不让公网、运行时或记忆凭证自动为内部业务数据背书。
5. 不改变知识库自身的空召回、引用校验和可选 NLI 反思流程。

## 4. 核心策略

### 4.1 审核结果从二态改为三态

事实审核输出以下三种结果：

- `PASS`：证据匹配，直接输出原回答。
- `PASS_WITH_WARNING`：证据缺失、类型不完全匹配或只能证明部分事实；输出原回答，并在正文中加入风险提示。
- `BLOCK_UNGROUNDED_FACTS`：保留为内部兼容枚举和观测结果，但 Main 用户输出路径不再用它替换正文或生成阻断卡片；统一转为 `PASS_WITH_WARNING` 展示。

事实取证门禁本身不再成为用户可见的硬阻断器。

### 4.2 风险级别

审核同时给出风险级别：

- `low`：存在同域可信凭证，只是类型不完全一致；例如知识库资料回答了文档口径问题。
- `medium`：存在可信工具结果，但与回答事实来源关联较弱，或者只能证明部分事实。
- `high`：没有有效凭证、工具执行失败，或回答包含实时金额、排名、库存、资产状态、运行状态等高风险动态事实。

所有级别均保留正文。风险级别只影响提示文案和观测指标，不触发事实阻断卡片。

## 5. 证据兼容策略

### 5.1 内部可信来源族

以下类型构成内部可信来源族：

- `internal_data`
- `internal_knowledge`
- `user_file`

当要求类型和实际凭证都属于该来源族时，允许兼容通过：

| 原要求 | 实际凭证 | 结果 |
| --- | --- | --- |
| `internal_knowledge` | `internal_knowledge` | `PASS` |
| `user_file` | `user_file` | `PASS` |
| `internal_data` | `internal_data` | `PASS` |
| `internal_knowledge` | `user_file` | `PASS_WITH_WARNING / low` |
| `user_file` | `internal_knowledge` | `PASS_WITH_WARNING / low` |
| `internal_data` | `internal_knowledge` | `PASS_WITH_WARNING`，实时业务数据标记为 `high` |
| `internal_data` | `user_file` | `PASS_WITH_WARNING`，实时业务数据标记为 `high` |

兼容通过只表示“回答存在内部可信来源”，不表示知识库或附件等同于实时数据库。

### 5.2 不兼容来源

以下情况不能降为低风险：

- `public_web` 为内部业务数据背书；
- `runtime_state` 为销售、订单、资产台账等业务事实背书；
- `conversation_memory` 为当前实时事实背书；
- 写操作成功为读取型事实背书；
- 失败、超时、拒绝、权限中断或错误结构被当作成功凭证。

这些情况输出原回答，但附加 `high` 风险提示并记录不匹配原因。

## 6. 请求要求合并

### 6.1 附件不再覆盖原要求

当前附件逻辑会直接把已有要求覆盖成 `user_file`。调整为：

1. 当前轮存在真实附件路径时，将 `user_file` 加入可接受类型，不覆盖 Router 已判定的来源。
2. 已有授权附件且用户明确说“继续看第二页/下一个 Sheet”时，加入 `user_file`。
3. 仅文本出现“文档、文件、表格”等词，但当前轮无附件且没有明确附件续问时，不强制要求 `user_file`。
4. 知识库与附件同时可用时，`internal_knowledge` 和 `user_file` 均为可接受来源。

### 6.2 UNKNOWN 请求

`UNKNOWN` 不提前绑定单一证据类型。最终根据以下信息生成风险提示：

- 本轮实际取得的凭证类型；
- 候选回答是否包含动态数值、执行声明、表格、实时状态或内部业务事实；
- 回答是否明确声明无法确认、示例、假设或模拟数据。

普通解释、代码、创作、翻译、方法和显式假设不提示风险。

## 7. 用户消息表现

### 7.1 不再发送阻断卡片

Main 最终输出不再发送用户可见的 `grounding_blocked` 事件，不再丢弃已生成正文，不再显示：

- “暂时无法验证事实”阻断卡片；
- “重新核实”按钮；
- “查看分析方法”按钮。

前端保留旧事件解析以兼容历史会话和旧后端，但新 Main 流程不再主动产生该事件。

### 7.2 正文风险提示

风险提示作为独立的 Markdown 引用块附加在回答末尾，避免打断阅读：

低风险：

> 信息来源提示：本回答基于知识库或已授权文件资料，不代表实时数据库状态。

中风险：

> 信息来源提示：本回答参考了已取得的资料，但部分结论未获得完全匹配的数据来源，请结合原始资料核对。

高风险：

> 风险提示：本次未取得能够完整验证这些具体数据或当前状态的来源，以下内容可能存在偏差，请勿直接用于生产操作或正式决策。

同一回答只追加一次提示。提示由服务端生成，模型不能自行取消或伪造审核结果。

## 8. 执行流程

1. 请求判源并生成事实证据要求。
2. 工具预检继续提示或促发匹配工具。
3. Runtime 工具成功后继续签发类型化 `EvidenceReceipt`。
4. Main 缓冲需要审核的候选正文。
5. 审核计算匹配结果、兼容关系、风险等级和原因。
6. `PASS` 输出原正文。
7. `PASS_WITH_WARNING` 输出原正文，再追加服务端风险提示。
8. 不产生用户可见的事实阻断卡片。
9. 中断恢复后的正文执行相同策略，避免恢复路径行为不一致。

## 9. 与现有门禁的边界

以下门禁继续硬阻断，不受本设计影响：

- 工具权限拒绝和角色禁用；
- `permission_required` 用户确认；
- `external_execution_required` 外部执行挂起；
- SQL 表范围、字段、权限、静态风险和安全校验；
- 文件路径授权和工作区隔离；
- 输入/输出安全审计；
- 敏感写操作和命令执行限制。

知识库 Runner 的空召回、引用有效性和可选 NLI 自反思仍保持原有领域规则。若知识库 Runner 最终安全熔断，其行为不由 Main 的柔性事实提示替代。

旧 `_is_hallucinated_data_reply()` 不再替换已经取得可信工具结果的正文。无工具且生成疑似内部数据时，也应改为保留正文并追加高风险提示，避免和平台级柔性策略产生两套用户体验。

## 10. 可观测性

每次事实审核记录结构化字段：

- `request_source`
- `required_evidence_types`
- `available_evidence_types`
- `decision`
- `risk_level`
- `reason`
- `warning_appended`
- `tool_attempted`

示例：

```text
request_source=internal_structured_data
required=[internal_data]
available=[internal_knowledge]
decision=pass_with_warning
risk_level=high
reason=internal_source_compatible_but_realtime_data_unverified
warning_appended=true
```

这些日志用于统计误判来源、风险提示比例和工具元数据缺口，不向普通用户展示内部枚举。

## 11. 测试与验收

### 11.1 应直接通过

- 知识库检索成功且回答带有效引用。
- SQL 成功并回答查询结果。
- 文件成功读取并总结文件内容。
- 普通解释、代码、翻译、创作、数学和显式假设。

### 11.2 应保留正文并提示风险

- 知识库检索成功，但 Router 误判为 `internal_data`。
- 当前轮存在附件，但实际有效内容来自知识库。
- 文件报告用于回答历史业务数据，但不能证明实时状态。
- 工具执行失败后模型仍输出具体数据。
- 没有任何凭证却输出实时排名、金额、库存、资产或运行状态。
- Bash 成功后模型输出内部销售数据。
- 专家直选缺少来源 hint，模型输出内部资产表格。

### 11.3 不得出现

- 新 Main 回答发送 `grounding_blocked` 用户事件。
- 因事实取证不足丢弃模型正文。
- 因事实类型不一致展示黄色阻断卡片。
- 同一回答重复追加风险提示。
- 任意工具成功后无条件标记为低风险。

### 11.4 回归范围

- 证据账本成功、失败和空结果策略；
- 工具别名证据类型解析；
- Main 首次生成和中断恢复；
- 子智能体共享账本；
- ChatBI 跨轮可信结果派生凭证；
- Knowledge Runner 自身引用和反思门禁；
- 前端旧 `grounding_blocked` 事件兼容解析。

## 12. 发布与观察

Main 事实取证审核统一采用柔性提示，不提供恢复用户可见阻断卡片的运行时开关，避免不同环境出现两套不可预测的交互。权限、安全审计、SQL 安全和凭证签发不受影响。发布后重点观察风险提示占比、知识库成功后提示比例、证据类型不匹配比例以及用户重试率。
