# 智能体主类型与委派就绪状态设计

## 背景

当前新建智能体直接要求用户填写 `capabilities` 编排标签。`data_query`、`knowledge_base` 等内部协议容易被误填成 `chatbi`、`dataquery` 或中文名称，导致智能体保存成功但无法被 Main 自动识别。

同时，Main 的可委派候选只校验基础智能体是否启用、是否为系统智能体及用户权限，没有验证本地智能体是否存在已发布版本、所需工具和资源绑定是否完整。这会造成候选清单显示可用，但 `sub_agent_call` 实际加载失败。

## 目标

1. 新建和编辑智能体时，以用户可理解的主类型替代内部能力标签自由输入。
2. 首期主类型固定为：通用助手、数据分析（ChatBI）、知识库助手。
3. 主类型单选；其他专业智能体统一归为通用助手。
4. 主类型自动维护不可删除的系统能力标签；高级用户仍可配置扩展能力。
5. 智能体允许先创建后配置，但只有满足发布就绪条件时才能发布并进入 Main 委派候选。
6. 兼容已有智能体及现有 `capabilities` 消费逻辑。

## 非目标

- 首期不支持管理员创建新的主类型。
- 首期不支持一个智能体同时拥有 ChatBI 和知识库两种主类型。
- 不在本次改造中重做完整的新建智能体多步向导。
- 不改变 `sub_agent_call(agent_name, query)` 的工具协议。

## 产品交互

### 首次创建一站式向导

首次创建采用同一弹窗内的三步向导：

1. **基本信息**：标识、名称、描述、主类型和引擎。
2. **初始版本**：模型、温度、系统提示词、工具和 Skills。
3. **资源与发布**：按主类型绑定数据集或知识库，展示就绪检查并决定是否发布。

第一步点击“保存并继续”后，系统立即创建智能体基础记录，并自动创建 V1 `DRAFT`。V1 内容来自对应系统模板当前已发布版本的快照：通用助手参考 `main`，数据分析参考 `chat-bi`，知识库助手参考 `knowledge-base`。快照创建后完全归属新智能体，后续系统模板升级不会覆盖用户草稿。

若参考模板不存在或没有可加载发布版本，后端使用平台默认模型、类型对应的最小安全提示词及必要工具生成草稿，并在响应中返回 `template_fallback=true` 供前端提示。

每一步均提供“保存草稿，稍后配置”。第一步使用该操作时仍同时创建智能体与 V1 草稿；第二步先保存版本内容；第三步先保存资源绑定。草稿及未就绪智能体不会进入 Main 路由或委派候选。

智能体卡片对未完成的首次配置显示“继续配置”，点击后根据持久化状态恢复到最早未完成步骤，不依赖浏览器本地状态。创建请求使用客户端生成的 `onboarding_key` 实现幂等；相同用户、相同 key 重试时返回已创建的智能体和 V1，不重复创建记录。

### 主类型选择

新建和编辑智能体弹窗展示三个固定卡片：

| 主类型 | 展示说明 | 锁定系统能力 |
| --- | --- | --- |
| 通用助手 | 问答、写作与专业任务 | `general_chat` |
| 数据分析（ChatBI） | 查数、指标与报表分析 | `data_query` |
| 知识库助手 | 企业文档检索与问答 | `knowledge_base` |

默认选择“通用助手”。切换类型时，系统移除旧类型对应的锁定标签，并添加新类型对应的锁定标签；用户配置的其他扩展标签保留。

### 高级能力设置

“高级能力设置”默认折叠：

- 系统能力以锁定标签展示，不允许手工删除或重命名。
- 常用扩展能力以受控选项展示，例如 `sql_generation`、`reporting`、`document_search`、`qa`。
- 自定义标签放在更深一层的高级入口中，统一执行去空格、去重和格式校验，不允许与系统能力标签冲突。

普通用户看到中文能力名称；内部标识作为辅助信息展示给管理员。

### 保存与发布反馈

- 创建阶段允许只选主类型并保存，按钮文案为“保存并继续配置”。
- 未满足资源要求时，详情页显示“尚未就绪”及缺失项。
- 发布动作执行强校验；不满足时阻止发布，并逐项提示缺少的版本配置、工具或资源。
- 三步向导最后展示统一就绪检查；满足条件时提供“检查并发布”，不满足时定位到缺失配置所在步骤。

### 各类型第三步差异

- 通用助手：不要求额外资源，直接展示版本摘要与发布检查。
- 数据分析：绑定数据集并确认查数工具。
- 知识库助手：绑定知识库并确认 `search_knowledge_base`。

## 数据模型与兼容策略

### 新字段

在 `ai_agents` 增加非空主类型字段：

```text
agent_type: GENERAL | CHATBI | KNOWLEDGE_BASE
```

数据库默认值为 `GENERAL`，API 使用同一枚举。`capabilities` 字段继续保留，作为现有路由、runner 和扩展能力的兼容接口。

增加创建幂等与向导状态字段：

```text
onboarding_key: nullable string, owner scope unique
onboarding_step: BASIC | VERSION | RESOURCE | COMPLETE
```

已有智能体迁移时根据当前配置计算状态：存在发布版本视为 `COMPLETE`；存在任意版本为 `RESOURCE`；没有版本为 `VERSION`。已有智能体不强制补建 V1，只有新的向导创建流程保证自动生成。

### 旧数据回填

迁移按以下优先级回填：

1. 包含 `data_query`，或系统智能体名称为 `chat-bi`：`CHATBI`。
2. 包含 `knowledge_base`，或系统智能体名称为 `knowledge-base`：`KNOWLEDGE_BASE`。
3. 其他：`GENERAL`。

回填后补齐锁定标签：

- `CHATBI` 必须包含 `data_query`。
- `KNOWLEDGE_BASE` 必须包含 `knowledge_base`。
- `GENERAL` 必须包含 `general_chat`。

保留原有扩展标签，不删除历史数据。名称只用于一次性迁移兼容，运行时不再通过固定名称判断主类型。

### API 规范化

创建接口接收 `agent_type`。服务端根据主类型规范化系统能力标签，不能只依赖前端处理。智能体创建后主类型不可修改；更新接口忽略传入的类型变更，只按原类型规范化系统标签。未知主类型返回 422；扩展标签统一 trim、去重并校验格式。

新增原子创建接口：

```text
POST /api/portal/agents/onboarding
```

请求包含基本信息和 `onboarding_key`。单个数据库事务完成基础记录与 V1 草稿快照；响应返回 `agent`、`version`、`onboarding_step` 和 `template_fallback`。现有 `POST /agents` 保持兼容，不自动创建版本。

第二、三步继续复用现有版本更新与智能体资源更新接口，并在成功后推进服务端 `onboarding_step`。发布成功后设置为 `COMPLETE`。

## 发布就绪校验

统一增加可复用的就绪检查服务，输出结构化结果：

```json
{
  "ready": false,
  "missing": ["published_version", "data_query_tool", "dataset_binding"]
}
```

规则如下：

### 通用助手

- 存在当前 `PUBLISHED` 版本。

### 数据分析（ChatBI）

- 存在当前 `PUBLISHED` 版本。
- 包含锁定能力 `data_query`。
- 发布版本绑定平台认可的查数工具。
- 数据集绑定可选；未显式绑定时使用当前用户已授权的数据集范围。

### 知识库助手

- 存在当前 `PUBLISHED` 版本。
- 包含锁定能力 `knowledge_base`。
- 发布版本绑定 `search_knowledge_base`。
- 智能体配置绑定至少一个可用知识库资源。

创建草稿不执行强拦截；发布操作执行强拦截。资源随后失效时，就绪状态实时变为不可用。

## Main 委派规则

Main 的动态通讯录和 `sub_agent_targets_by_capability` 必须使用同一个候选解析服务，候选必须同时满足：

1. `is_enabled = true`。
2. `is_system = true`。
3. 不是当前智能体。
4. 当前用户拥有智能体权限。
5. 存在可加载的当前发布配置。
6. 主类型对应的资源与工具处于就绪状态。

`data_query` 和 `knowledge_base` 仍作为编排协议，但不再依赖 `chat-bi`、`knowledge-base` 名称兜底。若同类型存在多个就绪智能体，首期按照明确的 `sort_order`、再按稳定 ID 顺序选择默认目标；所有候选仍进入通讯录，模型在用户明确点名时可调用指定候选。

工具执行阶段再次调用同一就绪检查，避免候选生成后资源状态变化。失败时返回具体缺失项，不再使用“未找到启用系统智能体”笼统描述，也不能把不可运行智能体重新列为合法候选。

## 首次创建交互（修订）

不再使用独立的“新建智能体”弹窗承载首次配置。复用现有 `AgentVersionEditorDrawer`，在原有版本配置步骤前增加“智能体信息”，形成统一五步抽屉：

1. 智能体信息：物理标识、显示名称、描述、固定主类型和基础属性。
2. 模型策略：沿用现有编排/合成模型配置。
3. 工具能力：沿用现有静态工具、MCP 和 Skills 配置，并根据主类型预选必需工具。
4. 系统提示词：沿用现有 Markdown 编辑器。
5. 确认保存：同时展示智能体信息与版本配置摘要。

从“新建智能体”进入时展示五步；从已有智能体“新建版本”进入时仍只展示原有四步。首次创建在最终确认时通过原子接口一次性创建智能体和 V1 草稿。用户在中途关闭时先通过相同接口保存当前智能体与 V1 草稿快照，再关闭抽屉；列表显示“继续配置”，重新进入后恢复到尚未完成的步骤。

原有独立基础信息弹窗仅保留给“编辑智能体元数据”，不再作为新建入口。资源绑定继续在工具能力页内通过既有工具配置按钮完成，不额外制造一套资源页。

### 引擎优先的自适应流程

“智能体信息”页首先选择执行引擎，再展示身份和引擎专属配置：

- `LOCAL`（NanZi Engine）：展示智能体类型、扩展编排标签，并继续模型、工具、提示词、确认四步。
- `RAGFLOW`：隐藏智能体类型和扩展编排标签，固定保存 `GENERAL/general_chat`；当前页完成 App、Dataset 和检索参数配置后直接创建，不创建本地版本。
- `OPENCLAW`：隐藏智能体类型和扩展编排标签，固定保存 `GENERAL/general_chat`；当前页完成地址、机器人、安全审计配置后直接创建，不创建本地版本。

外部引擎页面使用只读徽标说明其内置能力。内置能力不写入 `capabilities`，避免把产品说明词误当成 Main 的路由协议。切换到外部引擎时服务端提交前再次强制归一为 `GENERAL/general_chat`。

## 界面状态

智能体列表和详情展示：

- 主类型中文徽标。
- “已就绪 / 尚未就绪”状态。
- 尚未就绪时展示缺失项入口。
- 已创建但未发布的智能体不进入 Main 通讯录。

## 测试范围

### 后端

- 三种主类型与锁定标签的创建、更新和切换。
- 自定义标签规范化、重复和冲突校验。
- 旧数据迁移与标签补齐。
- 三种类型的发布就绪校验。
- 不可运行智能体不会进入动态通讯录和能力目标映射。
- 多个同类型候选按稳定优先级选择。
- `sub_agent_call` 对资源失效和配置加载失败返回准确错误。
- 一站式创建在同一事务创建智能体与 V1 快照。
- 相同用户和 `onboarding_key` 重试不会重复创建。
- 模板发布版本存在时复制快照，不存在时使用安全 fallback。
- 第二、三步保存后持久化推进向导步骤，退出重进可以恢复。

### 前端

- 默认选中通用助手。
- 类型卡片切换及系统标签同步。
- 锁定标签不可删除。
- 编辑旧智能体时类型正确回显。
- 尚未就绪提示和发布拦截错误正确展示。
- 五步首次创建抽屉的前进、后退、保存退出和断点续配。
- 已有智能体新建版本仍显示原有四步，不出现智能体信息步骤。
- 不同主类型在工具能力页预选并校验正确的必需工具与资源。
- 重复点击第一步不会生成重复智能体或版本。

### 回归

- 现有 ChatBI dispatcher、知识库 turn classifier、Main 动态通讯录及权限过滤测试继续通过。
- 已有 `capabilities` API 消费方保持兼容。

## 验收标准

1. 用户无需输入内部标签即可创建三类智能体。
2. ChatBI 自动包含 `data_query`，知识库助手自动包含 `knowledge_base`。
3. 配置不完整的智能体可以保存，但不能发布或被 Main 委派。
4. Main 展示的每个候选都能被 `sub_agent_call` 加载；状态变化后的失败信息明确可操作。
5. 多个同类型智能体的默认选择稳定、可解释。
6. 首次创建可在现有版本配置抽屉内完成智能体信息、V1 和资源绑定，也可任意步骤保存退出。
7. 新建智能体始终带有 V1 草稿，不产生空壳记录；网络重试不会重复创建。
