# 助手响应慢排查方案

> 基于 trace `bb9702ab-2da6-40e7-9918-7ad8b4a928f4`(总耗时 99435ms)的分析。
> 适用范围:nanzi 后端(`app/`,FastAPI + AgentScope),主模型 glm-5.2 经合思 AI 网关。
> 日期:2026-07-23

---

## 1. 现象

用户在助手发一条消息,端到端耗时约 **99 秒**。trace 四段:

| 阶段 | event_type | 模型 | 耗时 |
|---|---|---|---|
| 路由判断 | router (route_query) | deepseek-v4-flash | 2542ms |
| 主响应调用 | model_call (agentscope_model_call) | glm-5.2 | 18435ms |
| 智能体执行(父 span) | agent_execution | - | 47280ms |
| 总结生成 | synthesis | glm-5.2 | 18444ms |

四段相加 ~86.7s,另有 ~12.7s 间隙(序列化/网络/调度)。

**核心事实:一次对话触发了两次 glm-5.2 调用(主响应 + 总结),各 ~18.4s。**

---

## 排查结论(2026-07-23 已验证)

### 真实时间线(trace f70ebc3b,带时间戳)

| 阶段 | 耗时 | 说明 |
|---|---|---|
| router(跳过路由 LLM) | 3.5s | 路由 LLM 跳过了,但仍花 3.5s(非 LLM 处理) |
| 间隙1(router end -> intent) | 14.8s | 无 LLM 调用,黑盒 |
| intent(跳过) | 0ms | 问候短路 |
| 间隙2(intent end -> agent_reply) | 34.4s | 无 LLM 调用,黑盒 |
| glm-5.2 主响应 | 14.5s | 含首 token 7.8s |

### 核心发现:非 LLM 间隙占 78%,不是 glm-5.2

主响应阶段 67s 里:
- **非 LLM 开销 52.7s**(路由 3.5s + 间隙1 14.8s + 间隙2 34.4s)
- **LLM 调用 14.5s**(glm-5.2)

**之前判断"glm-5.2 慢是主因"被推翻**。真正大头是路由后的非 LLM 间隙(49s),这段时间没有任何 LLM 调用。"路由跳过 LLM 就该快"的直觉不对--跳过的只是路由那一步的 LLM,主响应 LLM + 中间 49s 非 LLM 处理照样要时间。

### 两个记忆子系统 bug(间隙嫌疑最大)

日志在间隙区间有两条 WARNING:
- `FT.CREATE failed: Cannot create index on db != 0` -- RediSearch 索引只能建在 db 0,但 `.env` `REDIS_DB=2`,记忆向量索引建不起来
- `Embedding failed: Embedding API 未配置` -- 记忆向量化失败

`memory_load` 阶段(`_load_memory_context`,内部调 `MemoryIndexService` / `DailySummaryService`)依赖 Redis + RediSearch + Embedding,在这两个失败下可能重试/超时,**最可能是 49s 间隙的大头**。

### 已加 4 个日志埋点(按现有 ChatStreamDebug 格式)

在 `app/services/ai/agent_service.py` 加了 4 个 `log_chat_stream_stage` 埋点,定位 49s 间隙:

| 埋点 | 位置 | 拆分的间隙 |
|---|---|---|
| `agent_resolved` | `_resolve_and_verify_agent` 返回后(权限校验完成) | 间隙1:权限校验耗时 |
| `memory_load` | `_load_memory_context` 前后 | 间隙2:LTM 记忆加载(最可疑) |
| `agent_roster` | 专家清单构建前后 | 间隙2:DB 查询 `resolve_delegable_system_agents` |
| `prompt_assembly` | `assemble_system_prompt` 前后 | 间隙2:系统提示组装(含 Skill) |

每个埋点带 `timestamp` + `elapsed_ms`,发消息后看日志(`[ChatStreamDebug]` 行)即可定位是哪段慢。

### 第二轮埋点结论 + 8 个黑盒埋点(2026-07-23)

**第一轮埋点结论(trace a53c28ac,端到端 72.3s):** 49s 间隙拆开后,`memory_load` 仅 1.78s、`agent_roster` 4.95s、`prompt_assembly` 4ms,都不是大头。真正大头是两段**未被第一轮覆盖的黑盒间隙**:

| 黑盒间隙 | 起止 | 耗时 | 夹的代码 |
|---|---|---|---|
| 间隙1 | `agent_resolved` -> `intent` | 13.0s | `setup_context` + `_inject_skills` |
| 间隙2 | `prompt_assembly` -> `agent_reply` | 26.0s | `_dispatch_executor` + runner `_execute_core`(工具加载 + agentscope agent 构建) |

非 LLM 间隙合计 54.9s(76%),LLM(glm-5.2) 17.35s(24%,含首 token 10.3s)。

**第二轮加了 8 个埋点拆这两段黑盒:**

| event_type | 位置 | 拆的段 |
|---|---|---|
| `ctx_setup` | `agent_service.py` `setup_context` 前后 | 间隙1:上下文初始化(DB 查知识库权限) |
| `skills_inject` | `agent_service.py` `_inject_skills` 前后 | 间隙1:skill 扫描/加载 |
| `dispatch` | `agent_service.py` `_dispatch_executor` 前后 | 间隙2:执行器调度 |
| `tools_resolve` | `runner._execute_core` `_resolve_runtime_tools_from_config` 前后 | 间隙2:工具加载 |
| `native_agent` | `runner._execute_with_agentscope_native_agent` 入口 -> `_build_native_agent` 前 | 间隙2:会话锁 `agentscope_session_lock.hold` + `agent_state_store.load` |
| `agent_build` | `runner._build_native_agent` 前后,fields 带 `load_ctx_ms`/`model_cfg_ms`/`workspace_ms`/`toolkit_ms` | 间隙2:agent 构建(含建主+fallback 两个模型 client) |
| `native_stream` | `agent.reply_stream` 前 | 间隙2:到首 token 最后一截 |

> 未单独埋点:`tools_resolve` 与 `native_agent` 之间的 tool-mode 准备(workspace sandbox 注入 + 多次 `ConfigService.get` + tool preflight `_resolve_available_sub_agent_delegation_info`),若日志显示这段间隙大再补。

### 下一步

1. 发消息,看新埋点的 `elapsed_ms`,定位 49s 具体在 memory_load / roster / assemble / 权限哪段
2. 若 `memory_load` 最大:修 `REDIS_DB=0`(或记忆索引单独用 db 0)+ 配 Embedding API
3. 若 `agent_roster` 慢:查 `resolve_delegable_system_agents_for_user` DB 查询
4. 若 `prompt_assembly` 慢:查 Skill 加载

## 2. 排查总流程

```
1. 复现 + 抓 trace(确认现象、拿 trace_id)
        ↓
2. 看 trace 四段耗时,定位最大段(用 spans 接口看嵌套)
        ↓
3. streaming 是否开启(影响用户首字感知,最高优先)
        ↓
4. glm-5.2 单次调用延迟(curl 直测合思网关,剥离后端)
        ↓
5. 总结生成(synthesis)是否同步阻塞用户响应
        ↓
6. 智能体执行内非模型开销(grounding / 上下文 / LTM)
        ↓
7. 路由开销(deepseek-v4-flash 能否短路)
        ↓
8. 基础设施(Redis 未启用的降级开销 / 网关延迟)
```

排查原则:**先剥离后端,直测模型网关**——确认是模型本身慢,还是后端处理慢。再按段定位。

## 3. 分项排查

### 3.1 复现与抓 trace

**操作:**
- 前端发一条消息,打开「执行链路追踪」(AgentDebug / trace 时间轴),记录 trace_id 与各段耗时。
- 后端日志:`logs/backend.log`,按 trace_id 过滤时间线。

**命令:**
```bash
# 按 trace_id 看完整时间线
grep -nE "{trace_id}" logs/backend.log

# 看本次会话的模型调用明细(输入/输出 token、耗时、是否命中 cache)
grep -nE "\[ModelCallStats\]" logs/backend.log | tail
```

**关键日志行(本案例):**
```
[Router] Routing decision found: agent_id='sys-agent-chat' confidence=0.95 ...
Creating AgentScope OpenAI-compatible model: model=glm-5.2 base_url=https://ai-gateway-prod.hosecloud.com/coding-plan/v1 key=apikey-6***
HTTP Request: POST https://ai-gateway-prod.hosecloud.com/coding-plan/v1/chat/completions "HTTP/1.1 200 OK"
[ModelCallStats] conv=... agent=main call#1 model=glm-5.2 in=7237 out=504 cache_in=7232 tools=False elapsed=18434.2ms
[ModelCallStats] conv=... agent=main call#1 model=glm-5.2 in=7237 out=624 cache_in=0   tools=False elapsed=22745.2ms
[SessionSummary] Updated summary for user=3 conv=...
```

**trace 数据接口(确认 span 嵌套关系,定位 47.3s 构成):**
```
GET /api/v1/chat/conversation/{conversation_id}/model_calls?trace_id={trace_id}
GET /api/portal/audit/traces/{trace_id}/spans
```
> 看 spans 的 `parent_span_id` 字段,确认 `model_call` 和 `synthesis` 是否嵌套在 `agent_execution` 内。
> 若嵌套:47.3s = model_call 18.4 + synthesis 18.4 + 其他 ~10.5(grounding/上下文)。
> 若并列:47.3s 里有 ~29s 非模型开销,需重点查 3.5。

---

### 3.2 streaming 是否开启(最高优先)

**为什么先查:** `get_llm_async` 默认 `streaming=False`。若主响应非流式,用户要等 504 token 全部生成完(18.4s)才看到第一个字;流式则首字快得多。这是用户感知延迟的最大杠杆。

**代码位置:**
- `app/core/llm/client.py`
  - `LLMFactory.get_chat_model(streaming=False)`(line 105,默认 False)
  - `get_llm_async(streaming=False)`(line 157,默认 False)
- `app/services/ai/runtime/agentscope/models.py:16` `create_openai_chat_model` 把 `config.streaming` 传给 `OpenAIChatModel(stream=...)`

**排查:**
```bash
# 找所有 LLM 创建调用点,看是否传 streaming=True
grep -rnE "get_llm_async\(|get_llm\(" app/ --include="*.py"
```
- 主对话流(chat completions)的调用点必须传 `streaming=True`。
- 若没传,主响应走非流式,用户等到全部生成完。

**判断标准:** chat 主流程调 `get_llm_async(streaming=True, ...)`。否则为问题点。

---

### 3.3 glm-5.2 单次调用延迟(剥离后端,直测网关)

**目的:** 确认 18.4s 是模型/网关本身慢,还是后端处理慢。

**操作:** 直接 curl 合思网关,测首字延迟 + 生成速度。

```bash
# 从 ai_models 表或模型配置页拿到 glm-5.2 的 api_key
# 非流式:测总耗时
time curl -sS -X POST https://ai-gateway-prod.hosecloud.com/coding-plan/v1/chat/completions \
  -H "Authorization: Bearer {API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"model":"glm-5.2","messages":[{"role":"user","content":"用 300 字介绍 Python"}]}'

# 流式:测首字延迟(time to first token)
time curl -sS -N -X POST https://ai-gateway-prod.hosecloud.com/coding-plan/v1/chat/completions \
  -H "Authorization: Bearer {API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"model":"glm-5.2","messages":[{"role":"user","content":"用 300 字介绍 Python"}],"stream":true}' \
  | head -c 200
```

**判断标准:**
- 后端日志 `out=504 elapsed=18434ms` → ~27 tok/s,偏慢。
- 若 curl 直测也是 ~27 tok/s → 模型/网关本身慢,后端无责。换更快模型或联系合思网关侧。
- 若 curl 直测快(如 100+ tok/s)但后端慢 → 后端有额外开销(序列化/重试/并发争抢),查后端。
- 流式首字延迟若 >2s → 网关排队或模型首 token 慢。

**对照模型:** 同样 curl 测 deepseek-v4-flash,对比速度,决定是否换主模型。

---

### 3.4 总结生成(synthesis)是否同步阻塞

**为什么查:** 每条消息都触发 `[SessionSummary] Updated summary`,多一次 18.4s 的 glm-5.2 调用。若同步(主响应返回前跑),用户白等 18.4s;若异步则不阻塞用户,但 trace 总时长仍记。

**代码位置:**
- `app/api/v1/endpoints/chat.py:619` `finalize_conversation` → `SessionSummaryService.finalize_session`
- synthesis span:`app/services/ai/runners/assistant_agent_runner.py:817 / 1419 / 1584`
- `app/services/ai/session_summary_service.py`(总结服务)

**排查:**
```bash
# 看总结调用与主响应的时序:总结是在主响应流内还是流后
grep -nE "SessionSummary|model_call_end|chat/completions" logs/backend.log | tail -40
```
- 若 `[SessionSummary] Updated` 出现在主响应 `chat/completions` 流结束**之后**,且前端已收到响应 → 异步,不阻塞用户(但占 trace 总时长)。
- 若出现在主响应**之前/之中** → 同步阻塞,需改异步。

**判断标准:** 总结应在主响应流式返回完成后,后台异步执行,不加入用户感知延迟。

---

### 3.5 智能体执行内非模型开销

**为什么查:** `agent_execution` 47.3s 减去 model_call 18.4s(再减 synthesis 18.4s 若嵌套),剩余 ~10-29s 是非模型开销,需定位。

**代码位置:**
- `app/services/ai/runners/assistant_agent_runner.py`
  - `execute`(line 522,agent_execution span 起止)
  - `_execute_raw`(line 530,grounding 审计、buffer 输出)
  - `_execute_core`(line 700,准备 LLM、构建 messages、模型调用)
- grounding 审计:`_execute_raw` 内 `GroundingService.audit`(line 650)

**排查:**
- 看 grounding 是否对**普通对话**也跑审计(`grounding_enabled` / `_should_buffer_grounding_output`)。若非数据查询也触发 buffer + 审计,会多开销。
- 看 `_resolve_runtime_tools_from_config`(line 716)是否每次都加载工具(可能涉及 DB/MCP 查询)。
- LTM 记忆检索:日志 `[LTM] Redis client not available for fetch_memory` —— Redis 未启用时的降级路径是否有等待/重试。

```bash
grep -nE "grounding|GroundingService|fetch_memory|LTM|_resolve_runtime_tools" logs/backend.log | tail -30
```

**判断标准:** 非数据查询场景不应触发完整 grounding 审计;工具解析应有缓存;LTM 降级不应有阻塞等待。

---

### 3.6 路由开销

**为什么查:** 每次非问候消息都跑 deepseek-v4-flash 路由判断 2.5s。问候已启发式短路,其他常见意图能否也短路。

**代码位置:**
- `app/services/ai/agent_service.py:656`(router span,route_query)
- 日志:`[Router] Routing decision found: ... reasoning='纯问候/寒暄，启发式短路至通用助手(跳过路由 LLM)'`

**排查:**
- 看路由的启发式短路条件,哪些意图能扩展短路(如明确的工具调用、简单查询)。
- 路由模型 = `llm_model_name` 配置项(DB `system_config` 表,当前值 `deepseek-v4-flash`);`get_llm_async()` 不传 model,内部读此项。base_url/key 不在 system_config(系统级为 None),而在 `ai_models` 表按 model_id 查每模型凭证(`config.py:107-140` model_registry lookup)。

```bash
grep -nE "route_query|启发式短路|Routing decision|Greeting shortcut" logs/backend.log | tail
```

**判断标准:** 路由 LLM 调用应尽量少;高置信度常见意图走启发式,省 2.5s。

**已查清(2026-07-23,trace a53c28ac router 3.5s):** 问候短路只跳过路由 LLM,短路前的两个 DB 查询照做:`_fetch_agents_from_db`(router_service.py:200,查全部 agents,60s 内存缓存) + `_filter_agents_for_user`(:209,按权限过滤)。本次是后端启动后首个 chat 请求,缓存为空冷查 + DB 连接池冷启动。**可优化**:应用启动时预热 agents 缓存;次要:查 `_filter_agents_for_user` 是否有 N+1 权限查询。

### 3.7 基础设施:Redis 与网关

**Redis(未启用的降级开销):**
- `.env` `REDIS_ENABLE=false`(本地无 redis-stack)。
- 日志多条 `Redis unavailable` / `Redis client not available`(MemoryService / LTM / ConversationRunLane)。
- 影响:记忆/会话锁/缓存降级,可能有同步等待或每次查 DB。
- **已修 bug:** `app/services/config_service.py:79` 原本 `redis=None` 时 `None.set()` 刷屏,已加 `if redis:` 保护(消除 `Failed to fetch config` ERROR)。

**排查:**
```bash
grep -nE "Redis unavailable|Redis client not available|ConversationRunLane|MemoryService" logs/backend.log | tail
```
- 评估启用 Redis(`REDIS_ENABLE=true` + 装 redis-stack)后,记忆/锁/缓存是否恢复正常,减少降级开销。

**合思网关延迟:**
- 网关 `ai-gateway-prod.hosecloud.com/coding-plan/v1`。
- 若 3.3 curl 直测也慢,联系合思网关侧确认:是否有排队/限流/跨地域延迟;glm-5.2 通道的实际吞吐。

## 4. 优化方向(按收益排序)

| # | 方向 | 预期收益 | 代价 |
|---|---|---|---|
| 1 | **确认/开启主响应 streaming** | 用户首字延迟从 18.4s → ~1-2s | 低(改调用参数) |
| 2 | **换更快主模型或修网关延迟** | 两次调用 37s → ~10s | 中(测模型/联系网关侧) |
| 3 | **总结生成改异步/降频** | 省 18.4s 同步阻塞 | 中(改 SessionSummaryService) |
| 4 | **路由启发式短路扩展** | 省 2.5s | 低 |
| 5 | **grounding 按场景跳过** | 省 ~10s(非数据查询) | 中 |
| 6 | **启用 Redis** | 恢复记忆/锁/缓存,减少降级 | 低(装 redis-stack) |

> 收益最大的是 1(streaming)和 2(模型速度),建议优先。3 取决于 3.4 的排查结论(若总结已异步则无需改)。
