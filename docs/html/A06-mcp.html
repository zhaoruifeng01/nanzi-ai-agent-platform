NanZi 开源智能体平台实战连载 · 06

MCP 在 NanZi 里怎么用

登记 · 双通道传输 · 发布进 Toolkit · 证据启发式

项目声明

本项目为个人开源（MIT）。仓库：RandyChen1985/nanzi-ai-agent-platform

MCP（Model Context Protocol）让外部系统以标准协议暴露工具。NanZi 不把每个系统写死进核心：登记 Server → 校验连通 → sync 缓存 → 发布 → 智能体版本勾选 → 进 AgentScope Toolkit。本篇按真实 API 与代码路径拆开。

代码入口：app/api/portal/endpoints/mcp.py · mcp_client.py · mcp_factory.py · 表 sys_mcp_servers / sys_mcp_tool_cache（V43）

01｜端到端流水线（先建立心智）

五步

1）系统配置登记 Server（name + sse_url + auth_headers）
2）创建/更新常自动 sync，也可手动
3）新工具默认未发布（is_published=False）
4）发布后出现在门户已发布目录，供版本勾选
5）运行时 Registry 按全名加载 → Factory → Toolkit → ReAct

常见翻车：sync 成功但对话没有工具 → 忘了 publish，或版本 tools JSON 没勾选；勾选了仍失败 → 查 SSE/Direct HTTP、鉴权头、对端超时。

02｜Portal API 全表（/api/portal/mcp）

POST /verify

干什么：临时连通测试，不落库。

场景：填 URL 前先验证网关是否活着。

原理：临时 session id=verify_{{uuid8}}，建 McpSseSession 后关闭。权限 element:system:config_save。

GET|POST /servers · PUT|DELETE /servers/{id}

干什么：列表/创建/改/删 Server。

场景：运维登记 Jira、GitHub、内部网关等。

原理：请求体字段：server_name、sse_url、auth_headers 默认 "{}"、enabled_status 创建默认 1。响应含 last_sync_at、tool_count、published_tool_count。创建/更新常自动 sync_tools。

POST /servers/{id}/sync · GET .../tools

干什么：拉远端 tools/list 写入缓存；查看缓存行。

场景：对端加了新工具后刷新平台目录。

原理：缓存表字段：tool_name（存全名）、tool_description、parameter_schema、is_published。schema 可带 x-nanzi-mcp-annotations / x-nanzi-evidence-types / x-nanzi-evidence-policy。

PUT /tools/{id}/publish?published= · POST /tools/{id}/execute

干什么：发布开关；可选试跑。

场景：灰度：先试跑再 publish；下线危险工具直接取消发布。

原理：只有 is_published=True 才会被 ToolRegistry 从 DB 加载。试跑走 Factory + ainvoke。

03｜传输：SSE 优先，Direct HTTP 回落

McpSseSession.connect()

1）优先标准 SSE：sse_client + ClientSession.initialize，超时 10s
2）失败 → is_direct_http=True，对 sse_url 发 JSON-RPC 2.0 POST
Direct HTTP：timeout 30s；protocolVersion 2024-11-05；clientInfo name=nanzi-ai-agent；顺序 initialize → notifications/initialized → tools/list|call
会话 id：响应头 mcp-session-id 或 body 实验字段；过期标记（session expired 等）+ HTTP 400/401/404/410 可恢复重连一次
空闲清理：60s 轮询，空闲 >300s 关闭会话

原理意义：生产网络与网关实现参差不齐；双通道让「标准 MCP」与「HTTP 网关」都能挂上，而不改业务工具代码。

04｜命名、发布与进 Toolkit

全名格式

同步时 full_name = server_name:raw_tool_name，例如 github:create_issue。Registry 查找用全名；调用远端时 split(":",1)[1] 剥前缀。

加载路径

版本 AIAgentVersion.tools JSON 列出名字 → ToolRegistry.get_runtime_tools → 静态注册表未命中则查 DB：McpToolCache 且 published → McpToolFactory.create_tool（source_type=mcp）→ build_toolkit → AgentScope Toolkit。DB 工具缓存 TTL 约 60s。

05｜只读启发式与 Grounding

两条路径

A. 注解：readOnlyHint=False → 关闭证据推断；True → 证据类型 external_tool，策略默认 allow_empty_success（可被 x-nanzi-evidence-policy 覆盖）
B. 名称启发式：取冒号后动作名；mutation 动词（create/update/delete/send…及中文）优先判非只读；否则需匹配 get/list/search/query… 才视为只读并记账 EXTERNAL_TOOL

实战建议：对会改数据的 MCP，publish 前确认注解；版本勾选配合用户 forbidden_tools；Grounding 开时只读查询更容易形成「有据」卡片。

06｜排障清单

对照表

连不通 → URL/网络/鉴权/SSE 是否被墙
列表空 → 是否 sync、对端 tools/list
对话无工具 → publish + 版本勾选 + Registry 全名一致
调用超时 → 对端与平台 30s；空闲会话是否过期
权限 → 配置菜单 element:system:config_save / menu:system:config
