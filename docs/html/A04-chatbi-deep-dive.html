深入 ChatBI

分诊 · 门禁 · Binding · 计划 · 栈 · 交付

本项目为个人开源（MIT）。GitHub

ChatBI 不是「让模型写一条 SQL」。实现是薄 DataAgentRunner + 厚 runners/chatbi/，外层仍是 AgentScope reply_stream，外面再套分诊、Binding、门禁、计划、自愈、结果栈与交付。

01｜分诊：DataQueryTurnType

轮次类型（内部最终判定）

new_data_query 新查数 · data_followup_query 追问查数 · metadata_query 元数据问答 · reuse_previous_result 复用结果 · result_analysis/result_presentation/result_action 结果分析/呈现/动作 · context_action · skill_execution · clarification_or_non_data / non_data_request / clarification_required · format_correction · federated_data_query 联邦查数
外层路由只负责选中 ChatBI；不是句句生成 SQL。复用/格式化可跳过 Agent 走 synthesis。

02｜SqlQueryBinding 与 SQL 门禁前缀

Binding

统一 Schema/SQL/元数据的表↔数据集↔列绑定，供预检与执行共用：含 sql、primary_dataset_name、schema_output、tables、preflight_validated 等。防「Schema 一套、执行另一套」。

门禁前缀（工具返回文案，不真执行）

[SCHEMA_GATE] · [SQL_REPEAT_GATE] · [SQL_STATIC_GATE] · [FAILED_SQL_REPEAT_GATE] · [SQL_PLAN_GATE] · [TIME_RANGE_GATE] · 沙箱 [Performance Blocked] · 预检 [TOOL_ERROR] SQL 预检失败
静态策略偏只读（SELECT/WITH/EXPLAIN/SHOW/DESC）；禁多语句与危险关键字；sqlglot 做方言校验。

03｜sql_plan 与自愈

计划 JSON 字段

goal · tables · fields · metrics · filters · time_range · grain · joins · risk_notes
build_platform_sql_plan 基于授权 Schema + 待执行 SQL；前端卡片；高风险可强制先过计划门禁。

repair

SQL 报错 / 空结果 / 缺计划 → repair_policy 决策 → 再开一轮 reply_stream。空结果结合意图帧诊断，避免无脑重写。

04｜结果栈 · 门户 · 交付

结果栈

chatbi_result_stack / data_result_stack_v1，push_result_ref 最大深度 10。可指「当前/上一个结果」或显式 result_id 下钻，条件继承。

数据门户

斜杠 /dataset_portal（兼容旧 /dataset_menu）；系统指令 id sys_dataset_portal；按授权生成数据集菜单 + LLM quick 追问。深链：dataset_portal=1&report_id&run_id 等。

简报与黄金报表

简报 chatbi_briefs：从结果栈沉淀证据化 brief（version2：title/facts/evidence/markdown），可导出 Word，portal /chatbi-briefs。
订阅表 portal_saved_report_subscriptions：schedule_type=daily|weekly|monthly|cron → cron_expr；ai_analysis_enabled；成功/失败通知；external_channels；alert_condition；投递审计 portal_saved_report_digest_deliveries。

边界：口径与元数据质量决定上限。排障先看 Trace：分诊类型、门禁前缀、Binding 表、修复轮次。Few-Shot 案例库值得运营投入。联邦跨库要治理。
