项目声明

「NanZi」来自我一直使用的网名（南孜），取「孜孜不倦」之意。仓库： RandyChen1985/nanzi-ai-agent-platform

本篇是连载开篇，目标只有一个：把 NanZi 真实跑起来的技术骨架讲清楚——用了哪些组件、请求怎么走、本地智能体为什么说「跑在 AgentScope 上」、企业侧 RBAC / 工作区隔离 / 工具授权与白名单 / Token 统计与额度如何卡住能力面与成本，以及 ChatBI / 知识库又在 Agent 之上叠了什么。

不会铺一堆「赋能」「闭环」空话。读完你应该能回答：用户敲下一句话之后，代码路径大致经过哪些类；谁能调哪个 Agent；文件落在谁的工作区；工具从哪注册进 Toolkit、审批与黑名单怎么拦；Token 怎么记账、额度超额怎么拦；RAGFlow / OpenClaw 为什么不走同一套 ReAct。

先给结论：自 2026-06 起，本地专家（通用 / 知识库 / ChatBI）的 AI 执行层已迁到 AgentScope 2.x（agentscope[service,storage,workspace]），以 Agent + Toolkit + ReActConfig 为唯一本地运行时，不再依赖 LangChain。平台侧有一层适配器 app/services/ai/runtime/agentscope/，文档里常称「AgentScope 运行时 / Runtime」——指的就是这套适配 + 上游 Agent 调度，不是另装一个无关产品。RAGFlow、OpenClaw 仍走各自 HTTP 代理，不经 AgentScope ReAct。

01｜产品定位：编排中枢，不是聊天壳

NanZi 做的是企业级多智能体编排与执行平台。对话只是入口；背后必须同时具备两件事——能编排专家干活，以及在权限与隔离约束下干活。缺后者，再强的 Agent 也只是演示壳。

强项：意图路由 + 多引擎专家（AgentScope / RAGFlow / OpenClaw）+ 企业 ChatBI + MCP·Skills·调度·Embed；以及贯穿全链路的 RBAC、用户/会话工作区隔离、工具与命令授权、Skills/API 白名单、Token 消耗统计与额度管理。
依赖：MySQL（平台库）+ Redis Stack（向量检索与调度锁）+ 自备 LLM 网关（OpenAI 兼容）+ 按需 RAGFlow 与业务库。
非目标：不是免运维公有云 SaaS；也不是「只调一个 completion API」的演示页。

02｜企业治理：RBAC · 隔离 · 授权 · Token 额度

这一节单独写，因为企业落地时，权限与隔离往往比「模型选哪家」更先卡住。NanZi 把治理拆成可组合的几层，而不是只靠一句 system prompt「请遵守权限」。

① 完善的 RBAC（不止登录角色）

采用 RBAC，分系统级与资源级，并支持菜单 / 元素级控制台权限：

· 系统角色：admin 跳过资源校验；user 只吃被授权资源。

· 业务角色 + 直接授权：用户可挂多个业务角色；资源类型覆盖 agent / dataset（知识库）/ metadata（ChatBI 元数据）/ api（V1 接口）等。

· 有效权限公式：admin OR 用户直接权限 OR 所属业务角色权限并集；聚合结果缓存 Redis（变更角色/直授立即失效）。

· 菜单 ≠ 数据：例如有 menu:knowledge_management 不等于能检索所有库；写操作还要过 element:knowledge:* 与 dataset 资源校验。

· V1 / Embed：API Key 调用时校验持有者对目标智能体（及白名单 API）的访问权，防止「有 Key 就能打任意 Agent」。

② 用户会话工作区隔离（文件系统级）

AgentScope LocalWorkspace 不是共享大盘。平台按身份生成稳定目录键 user_name__user_id，落在 agent_workspaces/{user_key}/ 下：

· docs/ — AI 默认落盘、跨会话共享（仅本人）

· uploads/ — 会话附件（仅本人）

· sessions/ — 按会话 UUID 隔离的会话沙箱

· skills/ — 个人技能目录（与平台公共 skills 分离）

文件浏览 / 预览 API 走 fs_access：普通用户只能进公共目录（如 branding、平台 skills）+ 本人私有工作区；越权路径直接拒绝。Admin 另议。这保证「读写文件 / Bash / 个人 Skill」不会串到别人的工作区。

运行时还有 会话锁（同用户同会话同 Agent 串行改 AgentState）与 Pending 快照按 user_id 落 Redis，避免并发把状态写乱、把确认串户。

③ 命令 / 工具授权：scope · 审批 · 黑名单

工具不是「挂上就能无脑执行」。运行时至少过三道闸：

· permission_scope：read 可直接放行；ask / 写类触发用户确认；dangerous 可直接 DENY。确认走 AgentScope PermissionEngine → SSE permission_required → 用户点确认后续跑。

· 智能体 approval_mode：可配置 allow / ask（默认）/ deny，覆盖整包工具的审批策略。

· 只读工具白名单：如 get_dataset_schema、search_knowledge_base、memory_search、get_current_time 等，始终自动 ALLOW，避免无害读操作弹确认轰炸。

· 用户级禁止工具 / 禁止命令：权限模型支持 forbidden_tools、forbidden_commands。运行时在工具调用前强制校验；命中命令黑名单（如限制 exec_command / Bash 类）直接拦截，理由可审计（如 hit_user_command_blacklist）。

④ 白名单式收敛：Skills · 工具挂载 · 数据与 API

· 智能体工具挂载：版本配置里勾选哪些工具 / MCP / Generic API 进 Toolkit——没挂上的，模型再「想用」也没有入口。

· Skills 白名单：开启 skills_custom 后，仅加载勾选的公共技能；未开则兼容「全部公共技能」。个人技能与平台技能目录隔离，禁用即不加载。

· ChatBI 数据权限：数据集授权 + Schema / SqlQueryBinding / 可选 data_perm 行级约束；数据门户只展示当前用户有权的数据集。

· 知识库：检索范围 = 角色/用户授权库 ∪ 自建 ∪ 默认公开库；有菜单权 ≠ 全库可读。

· V1 API 白名单：部分接口路径可配置白名单放行策略，与资源级 agent 校验配合，收紧对外面。

⑤ Token 消耗统计与额度管理

企业算力不是无限水龙头。NanZi 把「看得见」和「管得住」拆开做：

看得见：多粒度 Token 统计

· 会话级实时账本：AgentScope 侧挂 ModelCallStatsMiddleware，在 on_model_call 钩子记录每次 LLM 调用的模型名、输入/输出 Token（含缓存命中）、是否含工具调用、耗时；异步追加到 Redis List（键形如 nanzi:{uid}:{conv_id}:model_call_stats，TTL 与 AgentState 对齐约 7 天）。SSE 里也可带 model_call 事件，调试链路能对上每一次推理。

· 审计落库：执行历史写入 AgentExecutionHistory.total_tokens 等字段，供运营按用户聚合。

· 运营看板：控制台「Token 统计分析」按今日 / 7 天 / 30 天看总量、输入输出拆分、交互频次、智能体消耗分布，做算力审计而不是拍脑袋。

管得住：分层额度（Quota）

· 策略表 ai_agent_quota_policies，周期默认按月；用量来自当月执行历史 Token 汇总。

· 生效优先级（由高到低）：系统管理员绕过 → 用户专属额度 → 业务角色额度（多角色取更宽松的限额；角色可配「不限额」）→ 系统默认额度 → 未配置则不限额。

· 调用前拦截：AgentService 在对话入口走 QuotaService.check_before_call；超额返回 quota_exceeded，避免超限后仍烧模型。

· 管理面：Portal 提供系统 / 角色 / 用户额度 CRUD，以及「我的额度」查询（已用 / 限额 / 剩余 / 策略来源标签）。和 RBAC 同一套运营习惯：按人、按角色控成本。

治理心智图（记这一句就够）

谁能进控制台 / 调哪个 Agent靠 RBAC；文件与会话落在哪靠用户工作区隔离；这一刻能不能执行某工具或某条命令靠 scope + 审批 + 黑名单；智能体能力面收多大靠工具挂载与 Skills / 数据 / API 白名单；算力烧了多少、还剩多少靠 Token 统计与分层额度。五层叠在一起，编排才敢接企业生产。

① 接入层：两种入口，一套中枢

Embed Chat SDK：把助手嵌进业务系统（IFrame + postMessage，主题/水印/专家直选）。管理控制后台：Agent Studio、数据源与元数据、知识库、MCP、Skills、任务、RBAC、Token 额度等运营面。两端都经 SSE/HTTP 打进网关，而不是各玩各的后端。

② 核心网关（Portal Gateway）

架构图上的「网关」是逻辑中枢，实现上对应 FastAPI 的 /api/v1/*（对话与集成）+ /api/portal/*（运营）以及编排服务，而不是单独一个叫 PortalGateway 的类。四块能力：

· 统一鉴权：登录态 / API Key、RBAC 资源校验、额度预检

· 意图路由：直选 / 启发式 / LLM 路由 → Dispatcher

· 任务调度：APScheduler + Redis 锁，智能体身份周期任务与报表订阅

· 审计回溯：Trace、执行历史、Token 落库、敏感字段脱敏

③ 专家集群（Expert Pool）

网关选的是「哪类专家干活」：ChatBI（问数）、知识专家（RAG / Knowledge）、通用/插件助手（写作、工具链、通知、任务等）。产品上对应主类型 CHATBI / KNOWLEDGE_BASE / GENERAL，也可挂外部托管专家。多意图时可并行多个专家，再由合成模型聚合。

④ 运行引擎（Execution Engines）

专家「跑在哪套运行时上」是另一轴：

· AgentScope ReAct（LOCAL）：本地唯一 AI 运行时——思考/行动/观察、工具审批、ChatBI 自愈 SQL

· RAGFlow Agent：托管智能体 HTTP 流式代理，复用其对端检索与对话

· OpenClaw：安全网关代理，透传 AUTH_CONTEXT（身份、频道、可访问数据集）

⑤ 数据与插件层

引擎向下碰到三类「手脚」：企业多源数据仓/DB（Oracle / ClickHouse / MySQL / SQL Server 等，ChatBI 直查）；RAGFlow 知识库（非结构化文档）；MCP Server（外部系统与 API，工具缓存后挂进 Toolkit）。旁路还有平台 MySQL（配置/RBAC/审计）与 Redis Stack（状态、向量、调度、Token 会话账本）。

为什么拆 Executor / Runner / Runtime？

Executor（如 AssistantExecutor）极薄，只负责「这类请求交给谁」——对应专家池选型结果。

Runner（如 AssistantAgentRunner、DataAgentRunner）负责业务编排：拼消息、解析工具、消费事件、落 AgentState、ChatBI 守卫。

Runtime 适配层（runtime/agentscope/）把平台中立的消息 / 工具 / 事件，翻译成 AgentScope 的 Msg、Toolkit、reply_stream 事件，再映回现有 SSE JSON——升级 AgentScope 时，业务 Runner 与前端协议可以稳住。

主类型 vs 引擎（别混）

专家池里的主类型只有三个枚举：GENERAL / CHATBI / KNOWLEDGE_BASE，各自锁定主能力（general_chat / data_query / knowledge_base），发布与子智能体委派有就绪门禁。

引擎是另一轴：LOCAL（AgentScope）、RAGFLOW、OPENCLAW。同一工作室可混编——这正是上图「专家集群 × 运行引擎」二维结构的含义。

05｜AgentScope 运行时：我们到底怎么用

OpenSpec 写得很硬：本地 AI 执行链路 MUST 以 AgentScope 为唯一运行时，不得再走 LangChain Tool / Message / ChatOpenAI。NanZi 用的是上游的 Agent 调度器 + Toolkit + Permission + Workspace + Event，不是自己手写一套 ReAct while 循环。

一次本地专家调用的标准生命周期

① ToolRegistry.get_runtime_tools() → 统一的 RuntimeToolSpec[]（静态工具 / Generic API / MCP / 系统隐式工具）

② build_toolkit(specs) → AgentScope Toolkit（AgentScopeRuntimeTool 包装，可带审批）

③ 构造 Agent(name, system_prompt, model, toolkit, react_config=ReActConfig(max_iters=…), offloader=LocalWorkspace)

④ async for event in agent.reply_stream(inputs)：内部 while cur_iter < max_iters，在 reasoning / acting 间切换，推理前可触发上下文压缩

⑤ map_standard_agentscope_event() → 平台 SSE（content / log / thinking / permission_required …）

⑥ 正常结束 agent_state_store.save()；若 ASK 权限挂起则写入 Redis Pending，确认后再 reply_stream(UserConfirmResultEvent) 续跑

# AgentScope → 平台 SSE（节选）

TEXT_BLOCK_DELTA      → {"content": "..."}

TOOL_CALL_* / RESULT  → type:log + audit trace

REQUIRE_USER_CONFIRM  → permission_required

REQUIRE_EXTERNAL_EXEC → external_execution_required

THINKING_BLOCK_*     → thinking

MODEL_CALL_*         → model_call + Redis Token 统计

适配层里还有什么「平台胶水」

· 会话锁 agentscope_session_lock：同会话并发会毁掉 AgentState，必须串行。

· AgentState 持久化：跨请求恢复 ReAct 进度（cur_iter、context、permission_context）。

· LocalWorkspace：作为 offloader，工具大结果可落盘；工作区键按用户隔离。

· ModelCallStatsMiddleware：拦截模型调用，异步写 Redis 耗时 / Token（与第 02 节额度与运营看板同源）。

· 无工具短路：通用助手若未挂工具，可走 synthesis LLM 直出，不必起完整 Agent——省一轮调度开销。

三类本地 Runner 在 Agent 之上的差异

Assistant：标准 ReAct；工具预检促发调用；Skill 扫描；不绑 search_knowledge_base（知识走 Knowledge 路径）。

Knowledge：进入 ReAct 之前平台自动检索知识库，把召回注入提示词；空召回 / 无引用可拦截胡编。

ChatBI（DataAgentRunner + chatbi/*）：同一套 reply_stream，但外面套了 DataRunState、Schema 预取、SqlQueryBinding、SQL Gate、sql_plan、查数完成前拦截最终答、repair 再进一轮流——这是「企业问数」和「裸 Text-to-SQL Agent」的分水岭。细节第 04 篇展开。

06｜路由原理：直选 → 启发式 → LLM → 分发

编排质量一半在路由。RouterService + intent_service 的设计意图是：能短路就别烧路由模型；能直选就别猜。

① 专家直选（跳过 LLM 路由）

请求带 agent_id / 名称、Embed 专家模式、或对话 @提及 → direct_agent_selection=True。业务系统嵌入时应用这个，别把「该找谁」交给概率。

② 启发式短路（仍不调路由 LLM）

· 纯问候 → 通用助手

· 明显联网搜索句式 → 通用助手

· 上一轮是 ChatBI，且会话亲和性判定为 BREAK → 强制回通用助手（避免「问完数又闲聊」仍钉在数据专家）

③ ChatBI 会话亲和性 KEEP / BREAK / UNCERTAIN

KEEP：继续待在数据会话，启发式不拆。BREAK：明确离开问数。UNCERTAIN：灰区，交给 LLM 路由。这比「永远粘在上一个 Agent」稳得多。

④ LLM 语义路由

结合近几轮历史与智能体元数据，输出主 agent_id、可选 secondary_agents、置信度、轮次标签等。低置信或失败回落通用助手。

⑤ Dispatcher + 多专家合成

AgentDispatcher 按 engine_type 与能力选 Executor；ChatBI 内部分诊（新查数 / 结果分析 / 元数据 / 非数据澄清）发生在 Data 路径内部，不归路由器细拆。

若开启多专家且存在 secondary：并行 dispatch → 队列交错打 log → 合成 LLM（不是第二轮 ReAct）把多路结果收成一条流式答复。

07｜工具体系与平台护栏（概览）

AgentScope 的 Toolkit 只认从 RuntimeToolSpec 构建出来的工具。业务侧用平台自己的 @tool（非 LangChain），经 Registry 变成 Spec，再包装进 AgentScopeRuntimeTool。来源包括：静态 Python 工具、Generic API（配置型）、MCP 同步缓存、系统隐式工具（任务/时间等）。

工具能不能执行，不单看「有没有挂上」——还要叠第 02 节那套：permission_scope / approval_mode / 只读白名单 / forbidden_tools·forbidden_commands。挂载白名单决定能力面；黑名单与审批决定执行瞬间是否放行。ASK 范围会触发 PermissionEngine → SSE permission_required → POST /api/v1/chat/permissions/{id}/confirm 续跑。

ChatBI 为什么不能只靠 Prompt

· SqlQueryBinding：表归属、字段、数据权限在 execute 前后同一份绑定校验

· SQL Gate：未先取 Schema、危险 SQL、重复失败 SQL、性能沙箱等直接拦截，工具返回门禁文案而不是真执行

· sql_plan：高风险查询可要求先产出平台可审计的结构化计划（前端卡片）

· repair：SQL 报错 / 空结果可按策略再开一轮 reply_stream

· Grounding：工具成功写入证据账本，会话级可审计 PASS / 告警 / 拦截无依据回答（与 SQL 硬门禁互补）

09｜边界与诚实承诺

· 本地路径是 AgentScope ReAct；外部引擎是代理，能力受对端版本与网络约束。
· 权限再细，也依赖运营正确授权：角色配错、数据集误授，护栏挡不住「有权限的错误人」。
· ChatBI 准确率上限由元数据与授权质量决定，护栏防的是越权与胡写 SQL，不是替你补齐错误口径。
· Grounding / 知识引用拦截能降低无依据输出，不能承诺零幻觉。
· 开源可二次开发：优先扩展 MCP、Generic API、Skills、Prompt 版本；新 Runner 需遵守门控契约，而不是在业务里硬耦 AgentScope 对象。
