## 1. 连载定位与读者

| 维度 | 建议 |
|------|------|
| **系列名** | 《NanZi 开源智能体平台实战连载》（实战为主，每篇穿插关键技术原理） |

| **语气** | 公众号技术科普：先场景痛点 → 再能力拆解 → 最后动手路径；少堆砌术语，关键概念用「一句话 + 一张图」讲清 |
| **每篇体量** | 建议 2500–4500 字等价阅读量；结构：标题区 / 导语 / 3–6 个小节卡片 / 小结 & 下期预告 / GitHub CTA |
| **每篇必备** | 项目声明（个人开源 MIT）+ 仓库链接；与上/下篇的钩子；1–3 张实机截图位（先占位说明） |

### 1.1 行文原则（公众号友好）

1. **一篇只讲一个主问题**：读者读完能回答「我能用它干什么 / 怎么下手」。
2. **能力边界要写清楚**：适合什么、不适合什么，比夸功能更涨信任。
3. **概念对齐平台真实现状**：例如 RAGFlow / OpenClaw 是**引擎（Engine）**，不是 `GENERAL / CHATBI / KNOWLEDGE_BASE` 三类主类型。
4. **少用内部文件名**，多用产品语言；需要给开发者的细节放到「进阶」小框或下期。
5. **每篇结尾固定**：下期预告 +「点赞/在看」轻引导 + GitHub。

### 1.2 建议发布节奏

- 每周 1 篇（或隔周 1 篇），优先发完「总览 → 部署 → 建智能体 → ChatBI」前四篇，形成完整钩子。
- 工具 / MCP / Skills / 通知 / 调度可视为「平台能力中篇」。
- 第三方集成与 Embed 为「开放生态后篇」。
- 加餐篇（记忆、Grounding、提示词工厂等）穿插在中后段，避免前期过密。

## 3. 各篇规划详解

### A01｜平台总览：架构 · 功能 · 能力边界

**一句话卖点**：用一张能力地图，回答「NanZi 是什么、适合谁、不适合谁」。

**推荐标题**：企业级智能体平台长什么样？NanZi 能力全景与边界  
**备选**：Connect Data. Orchestrate Intelligence. — NanZi 开源智能体平台导读

**写作目标**

- 建立品牌与心智：个人开源、MIT、企业级编排中枢。
- 讲清架构分层，不陷入源码。
- 明确能力边界，为后续连载埋钩子。

**建议结构**

1. **开场痛点**：企业内部助手常卡在「会聊天不会查数 / 会查数不可信 / 接不上业务系统」。
2. **NanZi 是什么**：一句话定位 + 能力矩阵（对话编排、ChatBI、知识库、MCP/工具、Skills、调度通知、Embed）。
3. **架构鸟瞰**（可复用 README 架构图）：
   - Embed SDK / 管理后台
   - Portal 网关（鉴权、路由、调度、审计）
   - 专家池（ChatBI / Knowledge / 通用 / 外部引擎）
   - 执行引擎（AgentScope ReAct / RAGFlow / OpenClaw）
   - 数据与插件（业务库、RAGFlow KB、MCP）
4. **核心工作流**：路由 → 分发 → 执行 → 聚合（专家直选 / @提及 / 多专家合成）。
5. **能力边界（必写）**：
   - 强项：企业问数闭环、多引擎混合、运维级调度、嵌入集成。
   - 依赖：MySQL + Redis Stack；知识库/ChatBI 场景通常需 RAGFlow；业务库自备。
   - 非目标：不是纯 SaaS 多租户商业产品文档；不是无编排的「裸 LLM API」。
6. **连载导览**：列出后续篇目，邀请收藏。

**素材与配图**

- `docs/images/nanzi-platform-architecture-16x9.png`、overview / promo 图
- 界面快照：`docs/snapshot/`（对话、Agent Studio、知识库、Trace）

**注意点**

- 主类型只有三类：`GENERAL` / `CHATBI` / `KNOWLEDGE_BASE`；RAGFlow、OpenClaw 是引擎。
- 避免把「开源」写成「免运维」；部署篇再展开。

### A02｜安装部署与二次开发

**一句话卖点**：两条路径跑起来 + 二次开发该改哪里、不该改哪里。

**推荐标题**：从零装上 NanZi：依赖、部署与二次开发入口  
**备选**：Docker 离线包与源码联调：NanZi 部署实战手册（精简版）

**写作目标**

- 读者能选对部署方式并完成「能登录」。
- 说清依赖组件与可选组件。
- 给出二次开发扩展点地图（为 A05–A07、A10–A11 铺垫）。

**建议结构**

1. **你需要准备什么**
   - 必需：Docker（生产）或 Python 3.10+ / Node 18+（开发）；MySQL 8 utf8mb4；**Redis Stack**（向量检索 / 调度锁 / LTM）。
   - 按需：RAGFlow（知识库与部分 ChatBI 链路）；业务数据源（Oracle / ClickHouse / MySQL / SQL Server）。
2. **路径 A：Docker 生产 / 离线**（GitHub Releases tar → load → `start-nanzi-ai-agent.sh`；强调架构选 amd64/arm）。
3. **路径 B：源码开发**（`db-prod` 初始化 → 配置 `.env` → 前端构建 → 本地联调；**不在文中代跑 `./dev.sh` 口号过多**，点到「本地一键脚本由实施方执行」即可）。
4. **数据库迁移心智**：`db-prod/V*.sql` 版本化；升级要跑增量。
5. **二次开发入口地图**
   - 不改核心：MCP、Generic API Tool、Skills、Prompt 版本、场景模板、Embed。
   - 改核心：新 Runner / 新内置工具（指向 gating 契约思想，细节可链 GitHub docs）。
6. **常见坑 FAQ**：Redis 不是 Stack、跨架构镜像打错、未初始化管理员、RAGFlow 未配导致知识能力不可用。

**素材**

- `HOW_TO_INSTALL.md`、`docker/README.md`、`DEVELOPMENT.md`、`env.example`

### A03｜如何创建智能体：通用 / ChatBI / 知识库

**一句话卖点**：按主类型选对「专家人格」，配齐就绪条件再发布。

**推荐标题**：三种智能体怎么选？通用型、ChatBI、知识库创建指南  
**备选**：从草稿到发布：NanZi Agent Studio 上手

**写作目标**

- 讲清主类型锁定的主能力，以及扩展能力标签。
- 给出三类最小可用配置清单（就绪门禁）。
- 带过引擎选择入口（LOCAL 为主，外部引擎见 A10）。

**建议结构**

1. **创建前先选型**：场景对照表（写作问答 → 通用；经营问数 → ChatBI；制度 SOP → 知识库）。
2. **创建流程（产品视角）**：基本信息 → 版本向导（模型/提示词/工具/MCP/Skills/数据集）→ 调试 → 发布。
3. **三类详解**
   - **通用（GENERAL）**：工具预检、Skill 扫描、可挂通知/办公/HTTP 等；适合主助手。
   - **ChatBI**：锁定 `data_query`；需 schema/SQL 相关工具与数据集授权；可讲「数据门户」入口。
   - **知识库（KNOWLEDGE_BASE）**：绑定知识库 + `search_knowledge_base`；预检索与引用。
4. **就绪与委派**：未就绪不宜发布 / 不宜被 `sub_agent_call`（用产品语言解释「为什么按钮灰了」）。
5. **实操小案例**：各花 5 分钟建一个最小智能体并对话验证。
6. **下期钩子**：ChatBI 内部到底怎么想、怎么查、怎么自愈。

**素材**

- Agent 版本配置向导截图、三类能力对比表  
- 参考：`agent_types.py`、`agent_readiness.py`

### A04｜ChatBI 深度：逻辑 · 架构 · 流程

**一句话卖点**：企业问数不是「生成一条 SQL」，而是「授权 Schema → 护栏 → 执行 → 自愈 → 连续分析 → 交付」。

**推荐标题**：深入 ChatBI：从自然语言到可信 SQL 的完整链路  
**备选**：sql_plan、自愈与黄金报表：NanZi 问数引擎拆解

**写作目标**

- 成为系列「技术深度代表作」，但保持公众号可读性（流程图 + 场景故事）。
- 覆盖分诊、连续分析、数据门户、简报/黄金报表订阅的产品叙事。

**建议结构**

1. **故事开场**：业务同学问「本月华东签单同比？」——传统 BI vs ChatBI。
2. **总体架构**：元数据 / 案例 Few-Shot / Schema 门禁 / SQL 护栏 / 执行适配器 / 结果栈。
3. **一次提问的旅程**
   - 意图分诊：新查数 / 结果分析与呈现 / 元数据问答 / 非数据澄清或委派
   - Schema 与授权
   - （可选）结构化 `sql_plan` 卡片
   - 执行与空结果诊断、自动修复轮次
   - 追问下钻与条件继承
4. **可信与交付**：依据展示、业务简报（MD/Word）、钉选黄金报表、订阅与告警（与 A08/A09 交叉引用）。
5. **数据门户**：`/dataset_portal` 个性化导航与 quick 追问。
6. **能力边界**：口径依赖元数据质量；复杂跨库有联邦能力但需治理；幻觉靠护栏+门禁降低而非「绝对为零」。

**素材**

- Chat 调试 Trace、sql_plan 卡片、数据门户、黄金报表订阅界面  
- 设计文档可消化后转写，避免原文粘贴

**篇幅建议**：本篇可略长（系列核心），可拆「上/下」若单篇过载：  
- 上：链路与护栏；下：门户、简报、订阅与监控。

### A05｜内置工具：清单 · 用途 · 如何开发

**一句话卖点**：智能体的能力边界 ≈ 工具边界；NanZi 已内置「手脚」，也可无发版扩展。

**推荐标题**：内置工具箱大揭秘：用途一览与扩展姿势  
**备选**：从 ChatBI 到钉钉通知：NanZi Tool 生态导览

**建议结构**

1. **工具在 ReAct 里的角色**（思考-行动-观察，权限挂起/恢复一笔带过）。
2. **分类速览表**（公众号用分组卡片，不必一次列全 40+）：
   - 问数：`get_dataset_schema` / `execute_sql_query` / …
   - 知识：`search_knowledge_base`
   - 任务：创建/暂停/手动跑…
   - 通知：站内 / 钉钉 / 企微 / 邮件
   - 办公文档：Excel / Word 读写
   - 工作区与执行：读写文件、命令（沙箱语境）
   - 记忆：偏好与 `memory_search`
   - 协作：`sub_agent_call`
   - 其它：HTTP、时间、网页检索等
3. **动态工具**：MCP 工具名、`SysApiTool` 通用 API 工具（无代码接入）。
4. **如何「开发工具」三条路**（由易到难）：
   - 配 Generic API / MCP（推荐业务集成）
   - 写 Skill 约束调用方式（A07）
   - 平台内注册新 StructuredTool（进阶，链开发文档）
5. **安全提示**：黑名单、权限范围、勿把密钥写进提示词。

### A06｜MCP 使用指南

**一句话卖点**：标准协议接外部系统，同步工具缓存后挂到智能体即可用。

**推荐标题**：MCP 在 NanZi 里怎么用：接入、同步、挂载  
**备选**：SSE / HTTP 双模式：让企业系统成为 Agent 工具

**建议结构**

1. **MCP 是什么**（30 秒科普）+ NanZi 为何原生支持。
2. **控制台操作流**：登记 Server → 校验连通 → 同步/发布工具缓存 → 智能体版本勾选绑定。
3. **运行时注意**：会话失效恢复、只读工具与事实凭证（可轻提，深讲放 B02）。
4. **典型场景**：Jira / 内部 HTTP 封装 / 研发效能工具。
5. **排障**：连不通、工具列表空、权限不足、超时。

### A07｜Skills：创建与使用

**一句话卖点**：Skill = 可版本管理的「打法说明书」；平台公共技能 + 个人技能双轨。

**推荐标题**：Skills 实战：平台技能、个人技能与白名单挂载  
**备选**：SKILL.md 怎么写，助手才会「按你的方法做事」

**建议结构**

1. **Skill vs 提示词 vs 工具**：三者分工（策略 / 人格 / 手脚）。
2. **平台技能**：管理端 CRUD、启用停用、压缩包导入、调用统计。
3. **个人技能**：个人中心隔离目录、启用开关、创建后引导挂载。
4. **智能体白名单**：`skills_custom` 开启后只挂勾选公共技能。
5. **运行时如何触发**：口头「使用 XX 技能」、关键词扫描；主助手自动扫描。
6. **最小示例**：写一个「周报结构」或「故障排查清单」Skill（展示 frontmatter 字段即可）。
7. **边界**：Skill 不能替代授权与 SQL 护栏；别指望只靠文档消灭幻觉。

### A09｜任务调度

**一句话卖点**：APScheduler + Redis 分布式锁，支持「模拟智能体身份」周期执行。

**推荐标题**：任务调度：让智能体按计划替你干活  
**备选**：从 Cron 到黄金报表订阅：NanZi 任务中心场景集

**建议结构**

1. **为什么对话不够**：早报、巡检、订阅要「无人值守」。
2. **任务中心能力**：Cron、启停、手动跑、变量、指标、通知补全。
3. **场景清单**：
   - 每日经营简报
   - 指标阈值告警（联动黄金报表订阅）
   - 周期性知识同步类系统任务（概念层）
4. **与对话工具打通**：智能体可自己创建/管理任务（工具名用产品表述）。
5. **运维注意**：Redis 锁、失败重试与通知、时区。

---

### A10｜第三方智能体集成：RAGFlow / OpenClaw

**一句话卖点**：一个工作室里混编本地 ReAct 与外部托管大脑，并透传身份与数据权限。

**推荐标题**：接入第三方大脑：RAGFlow 托管智能体与 OpenClaw  
**备选**：LOCAL / RAGFLOW / OPENCLAW：三引擎怎么选

**建议结构**

1. **引擎 vs 类型**：先纠偏概念，再讲选型。
2. **RAGFlow 引擎**：托管 Agent、复用检索与流式；适合已有 RAGFlow 资产的团队。
3. **OpenClaw 引擎**：安全网关、`AUTH_CONTEXT`、数据集 ACL、租户隔离叙事。
4. **混合编排价值**：路由到不同专家；主助手委派。
5. **配置检查清单**与常见失败原因。
6. **边界**：外部引擎能力受对方版本与网络约束；平台侧做好超时与审计。

### A11｜第三方系统嵌入 NanZi

**一句话卖点**：业务系统不重建聊天，用 Embed + V1 API 接入现有鉴权。

**推荐标题**：把 NanZi 嵌进业务系统：Embed SDK 与 V1 API  
**备选**：IFrame + postMessage：企业助手嵌入手册（精简）

**建议结构**

1. **两种接入**：嵌入式聊天窗口 vs 服务端调 Chat Completions。
2. **Embed 握手**：就绪事件、INIT_CONFIG、上下文更新、发命令；主题与水印。
3. **身份**：API Key / 登录态 / profile；租户与权限隔离要点。
4. **直达专家**：`agent_id`、专家模式、绕过自动路由的场景。
5. **安全**：水印防截屏、SSO 开关、审计脱敏（点到为止）。
6. **最小接入示例**（伪代码级，完整以 `docs/md/embed_integration_guide.md` 为准）。
