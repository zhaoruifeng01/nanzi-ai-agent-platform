内置工具箱手册

每个工具干什么 · 什么场景用 · 背后原理

项目声明

本项目为个人开源（MIT）。仓库：RandyChen1985/nanzi-ai-agent-platform · 工具注册表：app/services/ai/tools/registry.py

智能体能力边界 ≈ 挂上了哪些工具。本篇不写空话，按注册表把平台静态内置工具逐项拆开：用途、典型场景、关键原理/约束。另两类「动态工具」——已发布 MCP、启用中的 Generic API（SysApiTool）——放在文末说明。

运行时原理（先记）：业务工具 → RuntimeToolSpec → AgentScope Toolkit → ReAct 调用。未在智能体版本勾选的工具，模型没有入口。勾选后还要过 permission_scope（read/ask/dangerous）、审批模式、用户级 forbidden 黑名单。部分工具成功调用会写入 Grounding 证据类型（内部数据/知识/网页/文件/记忆/运行时）。

01｜问数三件套（ChatBI 核心）

get_dataset_schema · 取授权范围内的数据集结构

干什么：按自然语言关键词检索元数据，返回表/字段 YAML（或等价结构），供模型写 SQL 前「看见」真实 Schema。

场景：「PUE」「机房能耗」「华东签单」——先摸有哪些表、字段叫什么，再生成 SQL。

原理：可走 RAGFlow 向量或本地元数据模糊搜；无结果会反馈「换更通用词」。ChatBI 门禁常要求先 Schema 后 SQL；证据类型 INTERNAL_DATA，空成功也可记账。

execute_sql_query · 只读 SQL 执行

干什么：在业务库执行模型生成的查询，返回结构化结果（行数/字节有上限，防把整库灌进上下文）。

场景：一切「查数、同比、明细抽样」；dry_run 可只验 SQL 不执行。

原理：工具层 + ChatBI 外部门禁（SqlQueryBinding、静态风险、重复 SQL、sql_plan 等）。sqlglot 校验；只允许 SELECT/WITH/EXPLAIN/SHOW/DESC 一类只读；禁止多语句与危险关键字。空结果仍可作「暂无数据」证据。

update_dashboard_context · 大屏/前端上下文联动

干什么：让对话驱动前端切换视角（机房、时间窗等），本身多返回成功占位。

原理：执行器拦截成功调用 → SSE context_update → 前端改状态。适合「边问数边控大屏」。

就绪提醒：CHATBI 类型至少要启用 get_dataset_schema 与/或 execute_sql_query，否则发布/委派门禁不过。

02｜知识库

search_knowledge_base

干什么：在 RAGFlow 知识库检索文档/手册/制度，返回片段供回答与引用。

场景：SOP、制度问答、说明书检索。Knowledge 专家就绪必须挂此工具并绑定 dataset_ids。

原理：按显式 dataset_ids 或智能体配置/默认库解析范围，并过权限。特征关闭时返回禁用错误。Knowledge Runner 还会在 ReAct 前自动预检索。证据 INTERNAL_KNOWLEDGE；空召回也可记账（「未找到」）。

03｜任务中心（对话里管 Cron）

以下工具对接 TaskCenter + APScheduler；创建时绑定当前智能体身份与用户。部分属系统隐式工具集，便于助手直接「帮你设个每天 8 点的任务」。

create_recurring_task

参数：name、cron（标准表达式）、prompt（每次执行时给 AI 的指令）、可选 notification_channels（portal/dingtalk/wechat_work/email）。场景：日报、巡检、周报推送。

get_my_tasks / cancel_task / start_task / pause_task / run_task_manually

列表当前用户任务；取消；恢复调度；暂停；立即手动跑一轮。证据上 get_my_tasks 属 RUNTIME_STATE。

04｜消息通知（读用户配置，不钓 webhook）

send_portal_notification — 站内铃铛/Inbox（title、content、level）
send_dingtalk_message — 钉钉（个人中心已配机器人）
send_wechat_work_message — 企业微信
send_email — 邮件（SMTP/收件人来自个人配置）

场景：任务完成摘要、告警、主动提醒。「请用户把 webhook 发给我」是反模式——工具实现故意从用户通知配置取凭据。

05｜Jira 内置集成

jira_search — JQL/关键词搜工单（证据 INTERNAL_KNOWLEDGE，空结果可记账）
jira_get_projects — 列项目
jira_create_issue — 建单（写操作，通常需审批）

场景：运维/研发助手「查缺陷、建工单」。凭据走环境/系统配置中的 Jira 连接，不是让模型记密码。更通用的外部系统优先用 MCP（第 06 篇）。

06｜工作区文件与命令（沙箱隔离）

路径校验落在用户私有工作区（docs/uploads/sessions/sandbox），防目录穿越。与 AgentScope 内置别名对应：Read/Write/Bash/Grep/Glob 等。

read_file — 读文件（offset/limit/tail）；证据 USER_FILE
write_file — 写文件（通常 ask）
search_text — 工作区内文本检索（类 grep）

场景：改配置草稿、读上传附件、在会话产物里搜关键字。

exec_command · Shell

执行命令前过危险命令黑名单；用户级 forbidden_commands 可再拦。证据 RUNTIME_STATE。场景：受控脚本、格式转换——生产务必配审批 + 黑名单。

list_process — 列进程
manage_process — 启停等（保护平台自身 PID）

场景：助手拉起的后台任务管理；误杀防护内置。

07｜Skills 相关工具

list_available_skills — 列出当前可见技能
read_skill_instruction — 读取某 skill 的 SKILL.md 正文
create_skills — 创建/落盘技能包

场景：模型决定「先看看有哪些打法」或「按某技能说明书执行」。主助手还有关键词自动扫描；细节见 Skills 篇。

08｜辅助分析与联网

sqlite_scratchpad

会话隔离的临时 SQLite：可 import JSON 成表再 SQL 清洗/透视，不污染业务主库。目录优先用户私有 sandbox。场景：对查数结果做二次聚合、多表临时 join。

directory_tree_navigator — 目录树浏览
code_syntax_linter — 代码语法检查
web_renderer_and_snapshot — Playwright 渲染/快照页面

fetch_static_web_url — 拉静态页内容（证据 PUBLIC_WEB）
web_search_baidu — 百度搜索
system_http_request — 受校验的 HTTP 请求（method/url/headers/body；URL 有安全校验）

场景：公网资料补充、调内部 HTTP API（无 MCP 时的轻量方案）。SSRF 等风险靠 URL 校验与权限收紧。

get_current_time

取当前时间（默认 Asia/Shanghai）。系统隐式工具；证据 RUNTIME_STATE。场景：写「今天/本周」类相对时间 SQL 或文案前校准时钟。

09｜记忆

update_user_preference / fetch_user_long_term_memory / delete_user_preference — 长期偏好读写删（Redis LTM Hash）
memory_search — 按会话摘要/历史向量检索（Redis Stack RediSearch）

场景：「记住我喜欢用表格」「上周我们讨论过的方案」。证据 CONVERSATION_MEMORY；空检索可记账。依赖 Redis Stack。

10｜专家协作

sub_agent_call

干什么：主助手把子任务交给另一个已就绪智能体，收回可交付正文。

原理：校验目标就绪与权限；默认超时约 120s、结果截断、同目标限流；子流程若弹出 permission_required / external_execution_required，委派模式无法代确认，需用户直接打开子专家。输出会清洗 sql_plan 标签防污染父上下文。

11｜办公文档 Excel / Word

excel_document_read（read）/ excel_document_write（ask）
word_document_read（read）/ word_document_write（ask）

场景：读上传表格做分析、生成 Excel/Word 交付物并给下载链接。路径限制在附件/工作区；写操作默认要用户确认。

12｜动态工具（不在静态表里，但同等进 Toolkit）

MCP 已发布工具

名形如 server_name:remote_tool，来自 McpToolCache 且 is_published=True。智能体版本勾选后加载。只读/写倾向可从命名启发式推断，影响 Grounding 外部工具凭证。详见第 06 篇。

Generic API（SysApiTool）

控制台配置 HTTP 接口 Schema → 运行时变成工具，无需改 Python。适合「公司内部一个 REST 就能当 Agent 手脚」。工厂：GenericApiToolFactory。

13｜怎么扩展 + 安全清单

由易到难：① MCP / Generic API → ② Skill 约束调用方式 → ③ 在 tools/ 用平台 @tool 写函数并 ToolRegistry.register（Docstring 会进模型可见描述）。

□ 版本只勾真正需要的工具（能力面白名单）
□ 写/危险操作走 ask；用户 forbidden_tools/commands
□ 密钥在系统配置/个人中心，不进 Prompt
□ ChatBI 再叠 SQL Gate，不只靠工具层
□ 需要事实审计时打开 Grounding，看证据类型是否匹配
