南孜智能体平台

Connect Data. Orchestrate Intelligence.

连接企业数据，编排智能专家，交付可控 AI

过去一年，很多企业都开始尝试把大模型接入客服、知识问答、数据分析和办公自动化。但真正走到生产现场时，问题往往不在“模型会不会回答”，而在“模型能不能安全地接触企业数据、能不能调用业务工具、能不能被审计和治理”。

南孜智能体平台正是围绕这个问题构建的企业级 AI 智能中枢。它不是单一聊天窗口，也不是孤立的知识库问答，而是一套把数据、知识、工具、权限、审计和智能体编排整合起来的平台底座。

一句话理解南孜

南孜智能体平台让企业能够用一个统一入口连接内部数据、知识库和外部系统，通过智能体路由与执行引擎完成问答、查数、工具调用、任务调度和审计追踪。

一、企业 AI 落地，真正难在哪里

当 AI 只用于通用问答时，接入一个模型就能看到效果。但企业场景更复杂：销售数据在数仓，客户资料在 CRM，制度文档在知识库，流程动作在工单、邮件、审批和报表系统里。AI 要产生价值，就必须理解上下文、找到正确数据、调用正确工具，并且不能越权。

这也是很多 AI 项目从 Demo 到生产会遇到的断点：能回答，但不一定能执行；能执行，但不一定可控；能查数，但不一定知道该查哪张表；能接知识库，但回答缺少引用和追溯；能接工具，但权限、确认、挂起恢复和审计都要重新补一遍。

南孜选择的路线，是把这些能力沉到统一平台里：前端负责交互和嵌入，后端负责路由和编排，执行层负责智能体推理、工具调用和门禁控制，数据层负责连接企业真实系统。

二、从聊天窗口，到企业智能体中枢

南孜的上层体验仍然是自然的对话。用户可以通过管理控制台、嵌入式 Chat SDK 或开放 API 进入平台，在一个对话窗口里提问、查数、上传附件、选择专家、触发快捷指令，甚至通过 @ 提及把问题直接交给特定智能体。

但在这个窗口背后，平台并不是简单地把问题转发给模型。它会先处理会话记忆、用户身份、权限上下文和智能体配置；如果用户没有指定专家，RouterService 会结合最近对话、智能体能力和当前问题做路由；随后 AgentDispatcher 将请求交给 ChatBI、Knowledge、Assistant、RAGFlow 或 OpenClaw 等不同执行器。

这意味着用户感受到的是一个统一的 AI 助手，而系统内部实际运行的是一组分工明确的智能专家：有的负责查数，有的负责知识检索，有的负责工具执行，有的负责外部托管智能体或安全网关对接。

深度交互式对话

平台支持极速流式响应、自动路由、专家模式、@ 提及直选和多专家协同。对普通用户来说，它像一个统一入口；对平台来说，它是一次完整的“路由、分发、执行、聚合”过程。

长期记忆与跨会话回顾

南孜不仅保存单轮对话，还提供长期记忆偏好注入和 memory_search 按需检索能力。用户问“我们之前讨论过什么”或“按我之前的偏好处理”时，系统可以在受控范围内回顾会话摘要和长期偏好。

灵活嵌入企业业务系统

通过 Embed Chat SDK，南孜可以嵌入企业已有门户或业务系统，对接现有鉴权体系，透传用户身份和权限上下文。AI 不必成为一个孤立入口，而是可以进入真实业务页面。

三、ChatBI：让业务人员用自然语言分析数据

在企业里，数据分析往往卡在两个地方：业务人员知道问题，但不会写 SQL；技术人员会写 SQL，但不一定理解每个业务问题背后的真实语义。南孜的 ChatBI 能力，就是把自然语言、元数据、数据源和 SQL 执行闭环连接起来。

用户可以直接问“本月各产品线销售趋势如何”“哪个区域增长最快”“帮我看一下异常波动原因”。平台会先根据权限和语义找到相关数据集，再注入表结构、字段解释、指标口径和案例经验，最后由 ChatBI 专家生成 SQL、执行查询、解释结果，并支持 CSV / Excel 导出。

为了让查数可用也可控，南孜在 ChatBI 链路中加入了 Schema 门禁、SQL 护栏、结果复用、错误自愈、空结果修复、可选 sql_plan 结构化计划和前端 SQL 计划卡片。也就是说，它不是“让模型随便写 SQL”，而是让模型在平台规则和企业数据边界内完成分析。

我的数据门户

对业务用户来说，最难的常常不是“怎么问”，而是“不知道有哪些数据可以问”。南孜提供“我的数据门户”，按用户权限生成可访问的数据集导航、业务说明和 quick 追问，让用户从业务场景自然进入数据分析，而不是从表名开始。

四、知识库：让企业知识可问、可引、可治理

企业知识通常沉淀在制度文档、SOP、项目资料、运维手册、产品说明和历史经验中。如果只是把这些文档丢给模型，很容易出现无引用回答、引用不稳定、召回不可控的问题。

南孜提供可视化知识库管理中心，支持文档树形管理、切片预览、召回测试、语义合并和生命周期审计。Knowledge 执行器会在 ReAct 前自动检索知识库，并把引用注入回答链路，帮助用户看到答案来自哪里。

对已有 RAGFlow 能力的企业，南孜也可以对接 RAGFlow 托管智能体与知识库，复用现有检索和流式对话能力。平台既能建设本地知识工作台，也能集成外部托管知识智能体。

五、工具生态：从回答问题到执行任务

企业 AI 的价值不止在回答，还在执行。南孜支持 MCP 工具生态和平台内置工具注册，可以把 Jira、邮件、GitLab、通知、数据 API、任务中心等外部能力接入智能体。智能体不只是“知道”，还可以在权限允许时“去做”。

平台内置工具预检机制，会根据用户问题和已绑定工具的相关度，引导模型主动调用工具。对于需要确认的工具，系统支持权限挂起和恢复：用户确认后，智能体可以从中断点继续执行，而不是重新开始整轮对话。

任务中心则让智能体能力从“即时对话”延伸到“周期执行”。通过 APScheduler + Redis，平台可以模拟智能体身份执行周期任务或一次性任务，适用于日报、巡检、告警通知、数据同步、自动摘要等场景。

六、架构上，南孜如何工作

南孜的架构可以概括为六层：用户入口、接入网关、编排核心、智能体执行、扩展引擎、数据与系统。每一层都承担清晰职责，也为生产环境的稳定性和可治理性预留边界。

入口与网关：统一接入，统一鉴权

Embed Chat SDK、管理控制台和开放 API 都通过 Portal / V1 API 网关进入平台。网关负责 API Key、登录态、RBAC、SSE 流式响应和审计追踪，为不同接入方式提供一致的安全边界。

编排核心：会话、路由、分发

AgentService 负责会话记忆、上下文裁剪、长期记忆注入、智能体配置解析和权限选项处理。RouterService 决定请求该交给哪个智能体，AgentDispatcher 再把请求送到对应执行器。

执行引擎：AgentScope ReAct 与多引擎协同

本地 Assistant、Knowledge、ChatBI 基于 AgentScope 2.x 运行，支持工具循环、事件映射、模型调用统计、权限挂起恢复和状态持久化。RAGFlow 与 OpenClaw 保持直连实现，用于托管知识智能体和大模型安全网关场景。

数据与系统：连接真实业务现场

底层可以连接 Oracle、ClickHouse、MySQL、Redis / RediSearch、知识库和外部业务系统。平台通过元数据、权限、工具描述和审计日志，让模型在可控上下文中接触企业能力。

七、安全、审计与可观测，是企业 AI 的底线

企业 AI 不能只追求“聪明”，还必须可控。南孜把 RBAC、SSO、审计日志、敏感字段脱敏、工具权限范围、SQL 护栏、Trace Timeline 和导出能力放进平台底座。

在对话过程中，用户可以看到决策链、工具调用、SQL 计划卡片、引用来源和结构化查询结果。管理员可以通过审计与 Trace 追踪关键行为，排查模型调用、工具执行和数据访问过程。

对嵌入式场景，平台还支持安全审计水印，可以将用户名、时间戳或自定义文案叠加到 Embed 窗口背景，降低截屏外泄风险。

Prompt Factory：让生产行为可运营

在企业环境中，提示词不是一次性文本，而是需要版本管理、草稿管理、灰度调整和审计追踪的生产配置。南孜通过 Prompt Factory 管理系统提示词和运营侧草稿，让智能体行为更加可控。

八、它适合哪些场景

南孜适合那些已经有数据、知识和业务系统沉淀，希望把大模型能力真正接入生产流程的企业。它既可以作为内部员工的 AI 工作台，也可以作为业务系统中的嵌入式助手，还可以作为面向数据分析、知识问答和任务自动化的统一中台。

面向业务团队

用自然语言查数、看报表、追问指标变化，从“找人写 SQL”变成“直接问业务问题”。

面向知识运营团队

把文档、制度、SOP 和经验沉淀到知识库，让回答带引用、召回可测试、内容可治理。

面向平台与开发团队

通过 API、MCP、工具注册、权限确认和任务中心，把智能体能力接入现有业务系统。

面向管理与安全团队

通过 RBAC、SSO、Trace、审计、脱敏和水印能力，让 AI 能力在企业安全边界内运行。

南孜的目标：让 AI 成为企业可运营能力

从单点问答到智能体中枢，中间隔着数据接入、知识治理、工具执行、权限控制、流程审计和持续运营。南孜智能体平台希望把这些复杂工程沉淀为统一底座，让企业不只是“接入大模型”，而是真正拥有可连接、可编排、可审计、可扩展的 AI 能力。
